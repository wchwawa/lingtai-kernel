PR1A Explainer — Replay recorded tool results before synthetic heal

Branch: codex/pr1a-tool-result-replay-20260615. Worktree: /private/tmp/lingtai-kernel-pr1a-tool-result-replay.

Conclusion

PR1A adds a replay-first safety net for pending tool-call healing. When the conversation tail has unanswered tool calls, the kernel now checks logs/events.jsonl for an already-executed real tool_result before inserting a synthetic “may or may not have happened” heal result.

This directly addresses the failure mode where a tool side effect/result was durably logged but the model-visible wire history was later healed with a placeholder, losing the real payload.

Root cause

A tool result can exist in two places at different moments:

  1. Durable event log: ToolExecutor logs raw tool_result rows to logs/events.jsonl immediately after tool execution.
  2. Conversation wire: the turn loop later appends model-visible ToolResultBlocks to ChatInterface.

If execution succeeded but the wire update/continuation path failed or was rolled back, later recovery saw only a pending assistant tool-call tail. The old behavior synthesized a placeholder without consulting the durable log. The real result was present, but not replayed.

What changed

Recovery helper

New tool_result_recovery.py reverse-scans <workdir>/logs/events.jsonl from the tail, bounded by bytes/events. It matches exact tool_call_id; if tool_name is present, it must match too. Newest match wins.

Central wiring

ChatInterface.close_pending_tool_calls() now accepts an optional lookup. For each pending call: recovered real block first; otherwise the existing synthetic fallback.

Agent/session attachment

BaseAgent passes its recovery callback to SessionManager, which attaches it to created, rebuilt, restored, and manually assigned chat interfaces.

Safety

Lookup failures never break healing. Replayed oversized results are spilled before append. New replay/miss/failure telemetry excludes raw result and tool_args.

Risks and boundaries

  • Bounded recovery: a matching event older than the scan window misses and falls back to the existing synthetic heal.
  • No semantic guessing: recovery only trusts exact IDs and exact tool names when present. It does not infer from tool args or result text.
  • No PR2A/PR3A scope: WorkerStillRunning poison recovery and daemon terminal-state classification are separate PRs.
