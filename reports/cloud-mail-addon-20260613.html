What this adds

A new curated MCP server, cloud_mail, that talks to a self-hosted Cloud Mail deployment (Cloudflare Workers; https://github.com/maillab/cloud-mail) over its REST API using httpx. It mirrors the IMAP addon's tool-surface model but is a REST client — not IMAP/SMTP. Inbound mail is discovered by polling and delivered to the host agent's inbox via the LICC v1 callback.

Auth & envelope

  • Envelope { code, message, data }: any non-200 code or HTTP ≥400 becomes a CloudMailError carrying only status/code/server-message — never request bodies or secrets.
  • Public token minted from admin_email/admin_password via POST /public/genToken; sent as the raw Authorization header (not Bearer).
  • User JWT minted from user_email/user_password via POST /login; used for /email/send. Handles both data.token and bare-string data shapes.
  • One automatic re-auth retry on a 401/403 before surfacing the error.

Polling & LICC

  • One daemon thread per account polls /public/emailList (descending, page size 50) every poll_interval seconds.
  • Per-account watermark (<agent_dir>/cloud_mail/<alias>/watermark.json) tracks the highest delivered integer emailId.
  • First poll seeds silently (records the high-water mark, pushes nothing) unless notify_existing: true. Later polls push one LICC event per new row, oldest-first.
  • LICC metadata: source: "cloud_mail", event_type: "email", account, email_id, compound_id, from, from_name, to, created_at.
  • allowed_senders filter is case-insensitive; filtered senders do not raise events but the watermark still advances so they never replay.

user_email/user_password/send_account_id are optional and only needed for send. If absent, check/read/search/polling still work via the public API; send returns a clear "add these config keys" error.

Risk notes & limitations

  • No real-network test. All tests use an in-process httpx.MockTransport emulating the documented Cloud Mail envelope/routes. Field names (receiveEmail, sendType, emailList row keys) were taken from the cloud-mail source clone; a live deployment should be smoke-tested before relying on send.
  • read scans recent pages. The public API has no by-id fetch, so read pages up to 5×50 recent rows to find the emailId. Emails older than that window return a clear "not found in scan window" error rather than the body.
  • send semantics are best-effort. sendType is hard-coded to "count" and both text and content are populated; HTML-vs-text handling on the server side (Cloud Mail derives preview/HTML from content) may need tuning against a real instance.
  • Attachments unsupported in this first pass — a request with attachments returns an explicit unsupported error (it does not silently drop them).
  • Polling, not push. Inbound latency is bounded by poll_interval (default 30s). The 50-row poll page can miss a burst of >50 emails between two ticks; raise poll_interval frequency or page size if that becomes a concern.
  • Secrets discipline: the client never logs tokens/passwords/Authorization headers; accounts status and add_user results are redacted. Errors carry only HTTP status / envelope code / server message.
