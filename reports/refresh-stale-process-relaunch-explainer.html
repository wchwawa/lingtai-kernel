Refresh stale-process relaunch + daemon lifecycle fix

Conclusion: PR #224 now fixes both layers of the dev-1 refresh incident: (1) the refresh watcher still has a conservative stale-duplicate cleanup fallback, and (2) the parent agent stop path now reclaims daemon executor pools and CLI child process groups before withdrawing heartbeat/lock liveness.

Observed production failure

During dev-1 recovery on 2026-06-04, mimo-1 had stale status/history, no heartbeat file, and refresh telemetry ended with twelve relaunch attempts failing because the CLI duplicate-process guard found an old process in the same workdir.

error: another lingtai agent is already running in /Users/huangzesen/work/projects/lingtai-dev/dev-1/.lingtai/mimo-1
  PID 72272: ... Python -m lingtai run /Users/huangzesen/work/projects/lingtai-dev/dev-1/.lingtai/mimo-1
  If this is a stale process, kill it first.
...
{"type": "refresh_failed_permanent", "attempts": 12}
Root cause found after the initial patch: the old agent process was not merely an arbitrary stale PID. It had just launched multiple CLI daemon emanations. Refresh/stop removed the heartbeat and released the lock while daemon ThreadPoolExecutor workers and external CLI process groups could still keep the Python interpreter alive in ps. The watcher therefore saw “dead” via heartbeat/lock, while the CLI duplicate-process guard saw “alive” via process table.

Patch layers

LayerChangeWhy it matters
Daemon lifecycleDaemonManager.shutdown_for_agent_stop() factors the existing reclaim machinery into a lifecycle-safe helper. It kills tracked CLI process groups, sets cancel events, shuts down emanation pools and the CLI ask pool, and waits briefly for both primary daemon futures and CLI ask futures to unwind during agent stop.Stops daemon-owned workers/subprocesses from keeping the old Python process visible after parent liveness is withdrawn.
Agent stop ordering_stop() calls _shutdown_daemon_runtime(..., reason="agent_stop") after joining the main loop but before session close, heartbeat stop, manifest write, or lock release.Heartbeat/lock remain present and fresh during daemon teardown, preserving the “heartbeat lifetime equals process lifetime” contract.
Watcher fallbackThe refresh watcher parses duplicate-guard stderr, validates the blocker is still a same-workdir lingtai run process, refuses to touch it if a fresh heartbeat exists, otherwise sends SIGTERM then SIGKILL and retries relaunch.Defense-in-depth for real stale same-agent blockers, while avoiding signals against unrelated or healthy agents.
TestsAdds daemon shutdown/stop-ordering/CLI ask-future regression tests; keeps the watcher generated-script test; makes one pre-existing daemon folder-creation test deterministic by preventing immediate mock completion from racing the “running” assertion.Pins both the root-cause fix and the fallback behavior.

Safety properties

RiskMitigation
Killing a healthy agentDaemon cleanup only touches resources tracked by the parent DaemonManager. Watcher cleanup only runs after the CLI duplicate-process guard rejects relaunch and only for a process whose command line still contains lingtai run + wd. A fresh heartbeat (<60s) blocks watcher cleanup.
Killing unrelated processWatcher PID is extracted from guard output, then revalidated with ps -p PID -o command= before any signal. Daemon process-group kills use PIDs already spawned and tracked by the daemon manager.
Stop path hangs foreverAgent-stop daemon cleanup is bounded: pool shutdown is non-blocking; future waiting uses a finite timeout and includes CLI follow-up ask futures as well as primary daemon futures. Explicit daemon(action="reclaim") remains non-blocking.
Silent actionDaemon cleanup logs daemon_lifecycle_shutdown with counts/errors; watcher cleanup logs structured refresh_watcher_stale_duplicate_* events.
