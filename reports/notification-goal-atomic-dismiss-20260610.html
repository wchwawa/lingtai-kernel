Human intent
Make notification channels explicit, dismiss system events atomically, and keep goal details in goal.json rather than the reminder.

What changed

1. Notification channel allowlist

notifications.py now treats .notification/ as an allowlisted filesystem surface.

  • Built-in channels currently include bash, btw, cron, email, goal, molt, nudge, post-molt, soul, system, and tool_loop_guard.
  • Dynamic MCP channels are allowed by prefix: mcp.*.
  • Unknown .notification/*.json files are ignored by notification_fingerprint() and collect_notifications().
  • Kernel helpers publish(), clear(), clear_with_result(), and dismiss_channel() reject non-allowlisted channels.
  • An exact-channel registration helper exists for future in-process producers: register_notification_channel(channel).

2. Atomic single-event dismiss for system.json

system(action="dismiss", channel="system") still preserves the old whole-channel behavior when no event selector is provided. New optional selectors allow removing only one logical event:

system(action="dismiss", channel="system", event_id="...")
system(action="dismiss", channel="system", ref_id="goal:current")
  • Only channel="system" accepts event_id / ref_id.
  • Matching events are removed from data.events; unrelated system events remain.
  • If the last event is removed, .notification/system.json is deleted.
  • No-match dismiss is a no-op: it logs the attempt and leaves the file untouched.
  • Stale-version protection remains unless force=true is supplied.

3. Protected goal channel + short system reminder

.notification/goal.json is now the protected source-of-truth channel for the active goal.

  • Generic dismiss of goal is refused, even with force, because dismissing a notification should not imply canceling the goal.
  • Canceling a goal means deleting .notification/goal.json; completing it means setting status to done/superseded or deleting/replacing the file.
  • The reminder is a short goal.reminder event inside .notification/system.json with body exactly:
Goal reminder: read .notification/goal.json and follow its instructions; see the goal manual under system-manual.
  • Important locked behavior: the reminder triggers only when .notification/goal.json exists, is active, and the agent has been IDLE past the configured delay.
  • If goal.json is absent, no reminder is generated.
  • Details/objective/criteria stay in goal.json; the reminder does not duplicate them.
  • Dismissing the generated system event only hides that reminder; it does not cancel the goal.
  • If goal.json is deleted or marked inactive/done while a reminder is already present, the next IDLE goal check clears the stale goal.reminder event.

Review notes

  • The allowlist intentionally ignores unknown files rather than treating them as model-visible notifications.
  • goal is protected separately from existing producer-owned-state guards; it is a valid notification channel but a source-of-truth file, not an ordinary mirror.
  • The goal reminder uses the same system-event surface as other kernel events, which keeps the notification channel set small while requiring the new atomic event dismiss path.
  • This PR does not merge or alter any external addon. MCP notification channels remain supported through the mcp.* prefix.
