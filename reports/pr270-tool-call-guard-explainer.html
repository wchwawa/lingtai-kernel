结论

已按 Jason 的方向做成一层很薄的形式化 guard：默认仍然全 pass，不改变当前 tool call 行为；但现在每次执行前都有稳定的 ToolProposal → ToolCallGuard → GuardDecision 插槽，后续可挂 functional checks，并能在 deny 时生成结构化 rejection tool result。

新增结构

  • ToolProposal：normalized tool 名、args、tool_call_id、tool_trace_id。
  • GuardDecision：allowed/check_name/reason/action/severity/metadata/checks。
  • ToolCallGuard：函数式 check chain；空链返回 default_allow。

执行语义

  • unknown-tool validation 仍先于 guard。
  • allow：记录 tool_call_approved 后照常 dispatch。
  • warn：照常 dispatch，并给 dict result 附 _advisory.type=tool_call_guard。
  • deny：不调用 handler，合成 ToolCallGuardDenied tool result，记录 tool_call_denied。

为什么适合 #309 后续治理

#309 的重复外发不再需要写成一次性特判。后续可以把“同一 turn / 短窗口内重复 external_message_send 从 warn 升 deny”做成一个普通 check；bash/file/system/MCP 也能复用同一条 guard 链。
