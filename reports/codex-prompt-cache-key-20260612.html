Codex Responses prompt_cache_key support

Narrow adapter change that opts Codex (ChatGPT-OAuth) Responses requests into the backend prompt cache with a stable, conservative default key — without sending the parameters Codex rejects.

Executive summary

ready for PRlingtai adapter only61 focused tests passed

Codex's /backend-api/codex/responses endpoint accepts an optional prompt_cache_key that opts a request into cross-request prompt caching, keyed by a stable string. This PR threads an optional prompt_cache_key through OpenAIResponsesSession and defaults Codex Responses sessions to a stable lingtai-codex:{model}:v1. The standard (non-Codex) OpenAI Responses path is unchanged unless explicitly configured.

Why a stable key

Codex prompt caching reuses backend-side compute for a shared stable prefix across requests that present the same prompt_cache_key. The kernel's Codex path is stateless — it replays the full canonical interface on every turn — so a stable key lets successive turns benefit from cached prefix work. The default is deliberately conservative: keyed by model so distinct models never share a cache namespace, with a :v1 suffix so the key can be bumped if the stable prefix shape ever changes. No system/tools hash is computed — that would be a broader refactor for marginal benefit, so it is intentionally left out.

What Codex rejects (and we never send)

sent prompt_cache_key

Accepted by Codex. Sent only when set; the Codex adapter defaults it to lingtai-codex:{model}:v1.

never sent prompt_cache_retention

Codex rejects prompt_cache_retention: "24h" with Unsupported parameter. Never emitted.

never sent cache_control

Anthropic-style content-block cache_control is rejected by Codex (Unknown parameter: input[0].content[0].cache_control). Not implemented for OpenAI/Codex.

What changed

1. Optional prompt_cache_key on the session

OpenAIResponsesSession.__init__ gains a keyword prompt_cache_key: str | None = None, stored as self._prompt_cache_key. When set, it is added to the Responses create() kwargs on all three send paths: OpenAIResponsesSession.send, OpenAIResponsesSession.send_stream, and CodexResponsesSession.send_stream.

2. Stable Codex default

CodexOpenAIAdapter._create_responses_session passes prompt_cache_key=f"lingtai-codex:{model}:v1" when constructing the CodexResponsesSession.

3. Non-Codex path unchanged

The standard OpenAI Responses path leaves prompt_cache_key at None (off) unless a caller explicitly constructs a session with one.

Representative Codex request kwargs

{
  "model": "gpt-5.5",
  "input": [ ... full replayed interface ... ],
  "stream": true,
  "store": false,
  "instructions": "system prompt",
  "tools": [ ... ],
  "prompt_cache_key": "lingtai-codex:gpt-5.5:v1"
  // note: no prompt_cache_retention, no cache_control anywhere
}

Risk & notes

  • low risk Additive: a single optional kwarg, off by default outside Codex. No change to overflow recovery, reasoning capture, tool conversion, or cached_tokens parsing.
  • The default key is model-scoped and versioned (:v1); bump the suffix if the stable prefix ever changes shape.
  • No system/tools-hash key — intentionally avoided to keep the change small; can be layered on later behind the same kwarg.
  • The Codex endpoint's cached_tokens reporting (already parsed from input_tokens_details.cached_tokens) is untouched and remains the way to observe cache hits.
