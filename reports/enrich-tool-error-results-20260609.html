Enriched Tool Error Payloads

Small kernel PR to make model-visible tool errors self-repair-friendly while preserving detailed durable logs.

Executive summary

ready for PRkernel-only77 focused tests passed

This change upgrades ToolExecutor error results from terse {status:"error", message:"..."}-style payloads into structured payloads that a later agent turn can debug locally. The durable log remains the forensic source of truth, but the model-visible result now carries enough context to repair the next attempt without first mining logs.

What changed

1. Common error payload builder

Added ToolExecutor._error_payload() and _enrich_error_payload(). Error results now include:

  • error_type and error_phase
  • tool_name, tool_call_id, tool_trace_id
  • tool_args and sorted arg_keys
  • elapsed_ms, retryable, and _tool_error_payload_version

2. Better validation errors

Unknown-tool validation failures now include validation_reason="unknown_tool" and available_tools, plus trace and argument context.

3. Better dispatch exceptions

Dispatch exceptions include exception_type and a bounded traceback_tail, making stack-location clues model-visible without dumping unbounded logs.

4. Tool-returned errors are enriched in place

If an MCP/tool itself returns status:error, its original fields are preserved and missing repair context is added via setdefault.

Important tail fix: parallel errors

During implementation review I found one remaining lossy path: the parallel future.result() exception and timeout branches logged enriched payloads, but errors_map stored only strings. The final model-visible result was then rebuilt as a generic ParallelToolError.

This PR fixes that by storing the enriched dict payload in errors_map and using the same stamped payload for both durable log and model-visible result. A regression test monkeypatches the parallel executor to force a future exception and asserts the log payload equals the model-visible payload.

Representative model-visible payload

{
  "status": "error",
  "message": "chat_id must be integer",
  "error_type": "telegram",
  "error_phase": "tool_returned_error",
  "tool_name": "telegram",
  "tool_call_id": "tg-err",
  "tool_trace_id": "tg-err",
  "tool_args": {"action":"send", "chat_id":"", "text":""},
  "arg_keys": ["action", "chat_id", "text"],
  "elapsed_ms": 0,
  "retryable": "unknown",
  "_tool_error_payload_version": 1
}

Risk assessment

  • low behavior risk No provider wire shape changes beyond richer dict contents already returned as tool results.
  • payload size Dispatch exception traceback is bounded to 4000 chars; existing spill logic still applies before model visibility.
  • compatibility Existing tool-returned error keys are preserved; new keys are added with setdefault.
  • observability Durable logs remain detailed and now align with model-visible payloads for the parallel future/timeout edge paths.
