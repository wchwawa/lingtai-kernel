Post-Molt 通知补丁：蜕壳之后，提醒它继续原本的工作

解决“蜕壳之后的智能体忘记自己刚刚为何蜕壳”——补丁在 _context_molt 和 context_forget 完成后，向 .notification/post-molt.json 写入一份提醒，把蜕壳的 reasoning / summary 与下一步的指令一同投递给新生的会话。

分支：feat/post-molt-notification　· 基准：main @ 2952771　· 通道：post-molt　· 关联渠道：molt（上下文压力警告，互不干涉）

1. 起因与目标

蜕壳（molt）会清空 wire 上下文并重建会话；新会话只有：

  • 被回放的那次 psyche(action='molt') 工具调用本身；
  • 它写在 args.summary 里的“给未来自己的简报”；
  • system/pad.md / system/lingtai.md（boot 钩子重新挂载）；
  • .notification/ 下的存活文件（系统状态，并非会话记忆）。

问题在于：智能体的 reasoning（“为什么我决定现在蜕壳”）以及触发此次蜕壳的 上层任务，在新一轮 turn 开始之后只能从 summary 第一段中隐约推断。如果触发蜕壳的是 系统强制路径（context_forget via warning ladder / aed / 外部信号）， 那么 reasoning 完全为空，蜕壳之后的智能体往往会“原地发呆”——不知道刚才发生了什么、 也不知道下一步该做什么。

设计意图 让蜕壳完成后，新会话的第一条系统通知 就是一句明确的提醒： 你刚刚 molt 过，下面的 reasoning / summary 就是你给自己留的便条，请继续。

2. 设计概要

2.1 通道（channel）选择

通知文件系统以 .notification/<channel>.json 为单位读写。现有 molt 通道由 base_agent/turn.py::_check_molt_pressure 拥有， 在上下文压力跌落阈值时会 无条件清空。如果复用 molt，蜕壳之后 压力骤降的下一 tick 就会把刚刚写入的提醒清掉。

决定 使用独立通道 post-molt，使两条信号互不干涉：
  • molt：压力警告，由 turn 循环按阈值发布 / 清除。
  • post-molt：蜕壳后提醒，由蜕壳本身在收尾阶段发布， 由智能体 自己 通过 system(action='dismiss', channel='post-molt') 清除。

2.2 时机与字段

发布点在两条蜕壳路径的“收尾阶段”——summary 已写入磁盘、但函数尚未返回 result。 这是唯一一个能同时拿到以下字段的位置：

字段来源说明
initiator常量 "agent"（_context_molt）或 "system"（context_forget）。
source常量 / 形参 智能体路径恒为 "agent"；系统路径透传 warning_ladder / aed / 外部信号名。
molt_countagent._molt_count新计数（已自增并写入 manifest）。
reasoningargs.get("_reasoning") or args.get("reasoning") 仅 agent 路径有；ToolExecutor 会把可见的 reasoning 替换为 _reasoning，但内部调用与测试可能用未加前缀的键，故两个键都接受。
reminderreasoning 优先；否则取 summary 第一非空行 智能体面对的“一句话提醒”——前端面板、portal 卡片可以直接用它。
summary_path_write_molt_summary 返回值 持久化失败时为 None，但 key 仍存在，避免智能体再去 probe。
tokens_before / tokens_after蜕壳计量 提供完整上下文，方便智能体在 pad 中做一句记账。

2.3 指令文本（instructions 字段）

You just completed a molt. Read system/pad.md, the latest summary under
system/summaries/, and the most recent human-channel messages to reconstruct
context, then continue the prior task. Once you have re-engaged, dismiss this
reminder with system(action='dismiss', channel='post-molt').

通知系统的 submit() 把 instructions 摆在 envelope 顶层， 与 data 平级，正是为这类“需要明确指令才能正确处理”的通知设计的。

  • agent 路径 · 带 reasoning：post-molt envelope priority=high； data.initiator == "agent"，reasoning 原样写入； instructions 包含 “post-molt”。
  • agent 路径 · 不带 reasoning：reminder 退化为 summary 第一非空行； reasoning 缺失或为空。

5. 风险与边界

兼容性
  • publish_notification 写文件采用 tmp + rename，对同时进行的 sync 是原子的；蜕壳本身已经在收尾阶段重置 _notification_fp，新写入的 post-molt.json 会在下一次 sync 自然进入 wire。
  • 没有引入新的 schema、没有改动任何蜕壳返回值（result_dict 与现有契约一致）。
  • 没有触动 molt 通道的发布 / 清除逻辑；压力警告行为不变。
已知边界
  • _publish_post_molt 是 best-effort——若 publish_notification 抛错，只会记 post_molt_notification_failed 日志，不会阻断蜕壳返回。这与现有 _write_molt_summary 的 best-effort 策略一致。
  • 智能体若长时间不 dismiss，post-molt.json 会持续存在于 wire 中——这是预期：直到智能体确认接续，提醒就应保留。
  • 本补丁不修改 psyche 工具 schema、不引入新 action；现有“dismiss”动作（system(action='dismiss', channel=...)）原生支持。

6. 后续可考虑（不在本 PR 范围）

  • 把 post-molt 也登记到 register_generic_dismiss_guard 的反列表中——目前它 不 在保护名单内，因此 generic dismiss 即可清除，这正是预期。
  • 前端面板（TUI / portal）可以根据 icon: 🌱 + priority: high 单独着色，让人类操作者一眼看出蜕壳后等待接续。
