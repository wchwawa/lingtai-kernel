MCP 陈旧连接自愈补丁

当 stdio 流悄悄断开之后 —— 让工具调用不再返回空错误，并能自我重连重试一次。

Issue Lingtai-AI/lingtai#104 分支 fix/mcp-closed-resource-restart 核心文件 src/lingtai/services/mcp.py stdio MCPClient

一、症状

一个被唤醒 / 恢复（revive / cpr）的智能体仍然挂着 Telegram 的 MCP 工具，但每一次工具调用都返回： 

{"status": "error", "message": ""}

事件日志里能看到异常类是 ClosedResourceError，消息为空，elapsed_ms=0。

  • 直接用 Telegram 包 / API 测试是好的 —— 说明配置与凭据都健康。
  • system(clear) / refresh 修不好这个“看起来还活着、其实流已经断了”的桥。
  • 只有 suspend → 等进程停 → cpr 重启子进程才能恢复。

二、三个根因

1. 空错误（blank error）

call_tool 里的 future.result() 没有任何 try/except。 anyio 的 ClosedResourceError 的 str(e) 恰好是空字符串， 于是它要么裸抛、要么被上层渲染成空 message。启动路径同理： _run_loop 用 self._error = str(e)，类名丢失， start() 抛出的就是 "MCP server failed to start: "。

2. 不会重连（no restart on stale）

call_tool 从不识别 ClosedResourceError。会话对象还在、 _loop.is_running() 还是 True，于是 is_connected() 返回 True —— 尽管底层 stdio 流早已死掉。这正是 refresh / clear 救不回来的原因：它们看到的是一座“还亮着灯的空房子”。

3. start() 会撒谎（start lies after restart）

start() 在 is_connected() 时直接早返回，而且从不清理已锁存的启动状态： _ready 事件一旦 set 就不再清，_error 留着旧值， close() 之后 _closed=True 永久挡住重启。 即便不 close，陈旧的 _ready / _error 也会让新的 start() 立刻返回 —— 根本没有真正重连，却“声称”自己连上了。

三、修复

两个静态助手

@staticmethod
def _format_exception(exc) -> str:
    cls = type(exc).__name__
    msg = str(exc).strip()
    return f"{cls}: {msg}" if msg else cls   # 空消息 → 仅类名，永不空白

@staticmethod
def _is_stale_resource_error(exc) -> bool:
    if type(exc).__name__ == "ClosedResourceError":   # 按类名匹配，无需 import anyio
        return True
    text = str(exc).lower()
    return any(m in text for m in
               ("closed", "broken pipe", "stream", "transport"))

真正的重连：restart()

把所有锁存的启动 / 会话状态清干净，让随后的 start() 不可能再撒谎：

def restart(self) -> None:
    self.close()
    self._ready.clear()        # 否则 start() 会以为自己已就绪
    self._error = None         # 否则 start() 会拿旧错误重新抛
    self._closed = False       # 解除 close() 留下的永久封锁
    self._session = None
    self._read_stream = self._write_stream = None
    self._loop = None
    self._thread = None
    self._stdio_cm = self._session_cm = None
    self.start()               # 真正重新拉起子进程

call_tool：陈旧即重启，重试一次

调用工具 → 抛出 ClosedResourceError → 是陈旧资源？ → restart() → 重试一次
情形返回
正常成功原样返回工具结果（happy path 完全不变）
陈旧错误 + 重试成功返回重试得到的正常结果
陈旧错误 + 重试仍失败 {status:"error", message:"ClosedResourceError: MCP session closed; restarted once but retry failed"}
非陈旧错误（不重启） {status:"error", message:"<ClassName>: <msg>"} —— 永不空白
关键不变量
最多重启一次、重试一次。成功路径零开销、零行为变化；失败路径永远带类名，不再有空 message。

四、范围与取舍

  • stdio 是上报的传输方式，因此 MCPClient 拿到完整的“识别 → 重启 → 重试”闭环。
  • HTTPMCPClient 仅复用 _format_exception 修掉连接期的空错误 —— 不加陈旧重启，避免在未复现的传输上过度改动（遵循 issue 的“don't overreach”）。
  • 按类名而非 isinstance 识别 ClosedResourceError， 因此无需把 anyio 变成硬依赖。
