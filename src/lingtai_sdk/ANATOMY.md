# lingtai_sdk

> **Maintenance:** see the `lingtai-kernel-anatomy` skill. **Coding agents** update this file in the same commit as code changes. **LingTai agents** report drift as issues (mail or `discussions/<name>-patch.md`); do not silently fix.

The **public SDK doorway** — a curated, import-light front door for building and embedding LingTai agents. It re-exports the two implementation packages (`lingtai_kernel`, `lingtai`) under one stable path and carries the *seed contracts* (runtime, capability-bundle manifest) that later PRs will implement. This is a foundation package: it ships shapes and a doorway, not a live runtime.

> **Human-facing docs.** The developer guide for this package — what the SDK is / is not yet, canonical vs legacy import paths, and runnable quick examples — lives at [`docs/sdk/README.md`](../../docs/sdk/README.md), with runnable example scripts under [`docs/sdk/examples/`](../../docs/sdk/examples/) (validated by `tests/test_sdk_docs_examples.py`). The full staged engineering log is [`docs/sdk/architecture-foundation.md`](../../docs/sdk/architecture-foundation.md).

> **What is an `ANATOMY.md`?** See the canonical convention at `src/lingtai/intrinsic_skills/lingtai-kernel-anatomy/SKILL.md`. This file follows the same 6-section template as every other anatomy in the tree.

## Directory shape

The public SDK is organized into subpackages, with **thin compatibility shims at
the old flat top-level paths** so every legacy import (`lingtai_sdk.native`,
`lingtai_sdk.bundle_registry`, `lingtai_sdk.capabilities`, the `*_tools` modules,
…) keeps resolving to the *same objects* as the new canonical module:

```
lingtai_sdk/
  __init__.py, _version.py, _compat.py, errors.py, types.py, sdk_skill.py   (top-level, unchanged)
  runtime/   __init__.py (compat surface) · contracts.py        [legacy: runtime.py]
  client/    __init__.py (compat surface) · facade.py           [legacy: client.py]
  guard/     __init__.py · bridge.py                            [legacy shim: guard_bridge.py]
  bundles/   __init__.py · contracts.py [legacy shim: capabilities.py]
             host.py [capability_host.py] · core.py [core_bundles.py]
             registry.py [bundle_registry.py] · native.py [native.py]
             file_tools.py · file_mutation_tools.py · communication_tools.py
             lifecycle_tools.py · mcp_tools.py · knowledge_tools.py
             skill_tools.py · bash_tools.py · avatar_tools.py
             psyche_tools.py · soul_tools.py    (each also a legacy top-level shim)
  assets/
```

`runtime` and `client` are **packages whose `__init__.py` IS the compat surface**:
because a directory and a same-named `.py` cannot coexist, the package wins and its
`__init__` re-exports everything the old flat module exported (`import
lingtai_sdk.runtime` / `from lingtai_sdk.client import query` resolve to the
package). Every other moved module keeps a real `.py` shim at its old top-level
path doing `from .<new>.<mod> import *` + `__all__` re-export. The shim layer and
the same-object guarantee are pinned by `tests/test_sdk_directory_shape.py`.

## Components

- `__init__.py` — the curated public surface. Eager kernel re-exports (`BaseAgent`, plus `types`/`errors` names); wrapper-backed names (`Agent`, `FileIOService`, `MailService`, `LoggingService`, `SearchService`, `VisionService`) resolve **lazily** via :pep:`562` `__getattr__` against `_LAZY_WRAPPER_EXPORTS`, caching into `globals()` on first access. `__all__` is the canonical export list; `__dir__` mirrors it. Stage-10/11/13 SDK-internal lazy exports also expose `LingTaiClient`, `LingTaiSession`, `QueryResult`, `open_session`, `query`, and the runtime-contract DTOs/ABCs from `runtime/contracts.py` without importing the wrapper. The `_LAZY_SDK_EXPORTS` map targets the **canonical** subpackage modules (`.bundles.native`, `.bundles.registry`, `.client`, `.runtime`).
- `_version.py` — `__version__`, resolved best-effort from the `lingtai` distribution metadata (`importlib.metadata.version("lingtai")`), falling back to `"0+unknown"` so import never fails over missing metadata.
- `types.py` — re-exports kernel public types (`AgentConfig`, `AgentState`, `Message`, `MSG_REQUEST`, `MSG_USER_INPUT`, `ChatSession`, `FunctionSchema`, `LLMResponse`, `ToolCall`, `LLMService`) under a stable path. Kernel-only; no heavy provider SDK is pulled.
- `errors.py` — `LingTaiSDKError` (SDK base), `NativeRuntimeConfigurationError` (raised by the default native factory when LLM config is partial/absent, *before* any agent is built), `NativeRuntimeStartError` (stage 14 — raised when `start()` fails *during* boot: factory/construction/`agent.start()` raised; the session rolls back to safe `PENDING`/no-agent first; the original is chained via `__cause__` and the message never echoes secrets), plus a re-export of kernel `UnknownToolError`. Import-pure (kernel `types` only).
- `_compat.py` — the machine-readable migration map: `Deprecation` dataclass, `DEPRECATIONS` tuple, `active_aliases()`, `migration_for()`. Drives the docs migration table and the same-object round-trip test.
- `client/facade.py` (legacy shim: `client/__init__.py` re-exports it; old `lingtai_sdk.client` path preserved) — **thin public client/session facade** (stage 10 + stage 11 + stage 12 polish + stage 15 event-semantics fix, import-pure): `LingTaiClient` wraps any supplied `Runtime` and offers `query(message, ...)` plus `open_session()`; `LingTaiSession` is a small wrapper over a live `RuntimeSession` for multi-message use (`send` / `events` / `text` / `close`, context-manager close); `text()` is a text-only convenience drain that discards non-text events from its returned string, so callers needing state/tool/usage/error/raw data should call `events()` directly; `QueryResult` carries concatenated `TEXT` event output plus the full `RuntimeEvent` tuple; module-level `query()` is a one-shot helper and `open_session()` starts a live session helper. **Stage-15 event semantics:** because `RuntimeSession.events()` is a *non-draining cumulative snapshot* (see `runtime/contracts.py`), `query()` now reads it **exactly once** after the lifecycle settles (`bundles/native.py`/`client/facade.py`) — it no longer extends with `events()` after `send` *and* again after `stop`, which double-counted the early events against a real (snapshot) session (the old fake masked it by draining on read). `LingTaiSession` keeps its **own read cursor** so `events()`/`text()` return only the events appended since the previous read (an incremental drain-like view) without mutating the session; `raw_session` still exposes the full snapshot. It owns no backend behavior: tests inject a fake runtime (now contract-conformant/non-draining), and the default native runtime is imported lazily only when no runtime is supplied. No wrapper `Agent`, provider SDK, Anthropic backend, or kernel turn-loop import/change occurs here.
- `runtime/contracts.py` — **runtime contract seed** (pure DTOs/ABCs, no kernel import): `RuntimeState`, `EventKind`, `RuntimeOptions`, `RuntimeMessage`, `RuntimeEvent` (with `state`/`text`/`error` constructors plus `tool_call`/`tool_result`/`usage` activity constructors — `runtime.py:110-128`), and the `RuntimeSession`/`Runtime` ABCs. Describes how a *future* live runtime is driven (options in, messages in, events out). **Stage-15 `events()` contract:** the `RuntimeSession` docstring now pins `events()` as a *non-draining, re-iterable cumulative snapshot* — every call returns **all** events so far, in order, and reading never consumes them; backends MUST NOT drain on read (a draining backend breaks `query()`), and incremental "only what's new" views are the caller's responsibility (cursor in `LingTaiSession`). `STATE` event `data['state']` is documented as a `RuntimeState` *value*, so backends map their own life-state onto the taxonomy. No live runtime here.
- `bundles/contracts.py` — **CapabilityBundle manifest seed** (pure DTOs, no `lingtai`/`lingtai_kernel` import): `BackendReplaceability`, **`SecurityDanger`** (`safe`/`caution`/`destructive`) and **`TransportKind`** (`native`/`in_process`/`stdio`/`http`) — public `str`-valued Enums acting as allow-lists for `SecurityPolicy.danger` / `TransportSpec.kind` (the fields stay plain strings, no breaking type change), `RoleFlags`, `CapabilitySurfaces`, `SecurityPolicy`, `TransportSpec`, `BundleManifest` (with `validate()`/`to_dict()`), `load_manifest()` — the inverse of `to_dict()`, parsing a plain-dict declaration back into a *validated* `BundleManifest` (reconstructs the nested frozen dataclasses + enum, raises `BundleLoadError` on unknown enum value / non-mapping block / failed invariant), and `proof_bundle()` — a harmless metadata-only synthetic bundle. Public schema + (de)serialization only; native privileged handlers stay in the kernel/wrapper. Imports only the import-pure `..errors` (for `BundleLoadError`). **Stage-7 strict validation:** `validate()` now also rejects an unknown `security.danger` / `transport.kind` (so a *directly constructed* manifest is checked too); `load_manifest()` reads role flags (`required`/`privileged`/`native_only`/`can_override`) as *real* booleans via `_strict_bool` (a present non-`bool` like `"false"`/`1` is rejected — `bool` checked before `int` since `True`/`False` are `int`s — an absent flag falls back to its default), and coerces `metadata` / `transport.config` via `_mapping_block` (a non-mapping list/scalar is rejected with a clear `BundleLoadError`, never half-coerced).
- `bundles/host.py` — **CapabilityBundle host boundary** (import-pure; imports only `.contracts` + `..errors`): a private `_BaseBundleHost` holds the shared contract — validate the manifest on registration, enforce the manifest↔handler contract per surface via `_check_contract` (every declared `surfaces.tools`/`surfaces.resources`/`surfaces.prompts` name has a callable handler, no handler is undeclared), store handlers, and expose three read-only surfaces (`invoke(tool, **kwargs)` / `read_resource(name)` / `read_prompt(name, **kwargs)`, each raising `BundleHostError` for an undeclared name). Subclasses differ only in **admission** (`_admit(manifest)`, called after `validate()`, before any handler is stored): **`BundleHost`** is the *non-native* host — it **refuses** `privileged`/`native_only` bundles and non-`in_process` transports (the unchanged guardrail that keeps `system`/`psyche`/`soul` out); **`NativeBundleHost`** (stage 7) is the *native-authority* host — it may host a `privileged`/`native_only` bundle, but **only** when explicitly constructed with the keyword-only `native_authority=True` (defaults `False`, so privileged hosting is never granted by accident) and **only** for a `native` transport, applying the *same* validation and per-surface contract. `proof_host()` wires `proof_bundle()` to a deterministic `_echo` handler; `native_privileged_proof_bundle()` + `native_proof_host()` are the privileged-side mirror — a harmless synthetic privileged/`native_only`/`native`-transport bundle with one deterministic `native_noop` tool, hosted only because the host carries explicit authority. No real privileged surface is named; the privileged `system`/`psyche`/`soul` bundles are deliberately NOT migrated here (stage 8).
- `bundles/core.py` — **the `system` / `psyche` / `soul` core bundle manifests + a native adapter shim (stage 8)** (import-pure; imports only `.contracts` + `.host` + `..errors`): the first deliberate contact with the privileged core surfaces, **as manifest contract + stub-injection seam only**. `system_bundle()` / `psyche_bundle()` / `soul_bundle()` each return a `BundleManifest` built by the shared `_core_manifest` helper with the identical privileged posture — `required=True`, `privileged=True`, `native_only=True`, `backend_replaceability=NATIVE_ONLY`, `transport.kind == native` — declaring exactly one public tool named after the bundle (`system`/`psyche`/`soul`) and helpful, **non-secret** metadata (`core`/`role`/`actions`). They differ only in declared `SecurityDanger`: `system` is `destructive` (lifecycle control incl. irreversible teardown), `psyche` is `destructive` (identity/context control includes the set-once `name.set` true-name write), `soul` is `caution` (inner-voice consultation plus persisted flow/voice config). `core_bundle_manifests()` returns the three in stable order (system, psyche, soul); `core_bundle_names()` / `is_core_manifest()` are the matching helpers. The adapter shim `native_core_host(manifest, handler)` turns a core manifest plus a single **injected** callable into a `capability_host.NativeBundleHost` constructed with `native_authority=True` (refusing a non-core manifest or a non-callable handler with `BundleHostError`, then deferring manifest/handler-parity + native-authority/transport enforcement to `NativeBundleHost`); `native_core_hosts(handlers)` builds `{name: NativeBundleHost}` for all three from a `{name: callable}` mapping, refusing a missing or undeclared/extra handler (so a partial wiring can never boot a subset of the privileged core silently). **It migrates no implementation:** it never imports or calls the real `system`/`psyche`/`soul` — handlers are supplied by the native runtime in a later stage; importing `core_bundles` pulls in no `lingtai` wrapper module (asserted by `tests/test_sdk_core_bundles.py::test_core_bundles_import_is_pure_and_migrates_nothing`).
- `bundles/file_tools.py` — **the low-state `read` / `glob` / `grep` file-tool bundle manifests + an in-process host seam (stage 3A)** (import-pure; imports only `.contracts` + `.host` + `..errors`): the non-privileged, in-process mirror of `bundles/core.py` and the migration **template** for low-state file/query tools. `read_bundle()` / `glob_bundle()` / `grep_bundle()` each return a `BundleManifest` built by the shared `_file_tool_manifest` helper with the identical low-state posture — `required=False`, `privileged=False`, `native_only=False`, `backend_replaceability=REPLACEABLE`, `transport.kind == in_process` — declaring exactly one public tool named after the bundle (`read`/`glob`/`grep`) and helpful, **non-secret** metadata (`file_tool`/`role`/`actions` + a copy of the tool's argument `schema`, a language-neutral mirror of `lingtai.core.{read,glob,grep}.get_schema`). All three are read-only / query surfaces, so all declare `SecurityDanger.SAFE`. `file_tool_manifests()` returns the three in stable order (read, glob, grep); `file_tool_names()` / `is_file_tool_manifest()` are the matching helpers. The host seam `file_tool_host(manifest, handler)` turns a file-tool manifest plus a single **injected** callable into a `capability_host.BundleHost` (refusing a non-file-tool manifest or a non-callable handler with `BundleHostError`, then deferring manifest/handler-parity + non-privileged/`in_process` enforcement to `BundleHost`); `file_tool_hosts(handlers)` builds `{name: BundleHost}` for all three from a `{name: callable}` mapping, refusing a missing or undeclared/extra handler. **It migrates no implementation:** it never imports or calls the real `read`/`glob`/`grep` (which bind to the wrapper's `agent._file_io`/`agent._working_dir`) — handlers are supplied by the wrapper bridge `lingtai.core.file_bundle`; importing `file_tools` pulls in no `lingtai` wrapper module (asserted by `tests/test_sdk_file_tools.py::test_file_tools_import_is_pure_and_migrates_nothing`). Unlike the privileged core bundles, the non-native `BundleHost` *accepts* these directly. The wrapper-side bridge (which proves the bundle-execution pattern against the real behavior) is covered by `tests/test_file_bundle_bridge.py`.
- `bundles/file_mutation_tools.py` — **the side-effecting `write` / `edit` file-mutation bundle manifests + an in-process host seam (stage 3B)** (import-pure; imports only `.contracts` + `.host` + `..errors`): the side-effecting counterpart of `bundles/file_tools.py`. It lives in its own module rather than extending `file_tools` precisely because `file_tools` carries an invariant stage 3B must break — *every* `file_tools` manifest is read-only / `SecurityDanger.SAFE` via the shared all-`SAFE` `_file_tool_manifest`; write/edit mutate the filesystem and do **not** share one danger posture, so a sibling module keeps `file_tools` "all SAFE" and gives the side-effect posture an explicit home. `write_bundle()` / `edit_bundle()` each return a `BundleManifest` built by the shared `_file_mutation_manifest` helper with the same low-state posture as the read tools — `required=False`, `privileged=False`, `native_only=False`, `backend_replaceability=REPLACEABLE`, `transport.kind == in_process` — declaring exactly one public tool named after the bundle (`write`/`edit`) and **non-secret** metadata (`file_tool`/`side_effect: True`/`role`/`actions` + a copy of the tool's argument `schema`, a language-neutral mirror of `lingtai.core.{write,edit}.get_schema`). Unlike `_file_tool_manifest`, `danger` is a **required, per-tool** argument: **`write` declares `SecurityDanger.DESTRUCTIVE`** (creates or *silently overwrites an existing file wholesale* — an irreversible clobber, the same posture as the privileged `system` bundle) and **`edit` declares `SecurityDanger.CAUTION`** (a *bounded in-place* string replacement that refuses ambiguous edits — lasting but not a wholesale clobber, the same posture as `psyche`/`soul`). The danger posture is a **declaration only** — no guard is installed and live dispatch is unchanged; its observable consequence is what the stage-17 `guard_bridge` derives from these manifests (`write` denied in `BLOCKING` / warned in `ADVISORY`; `edit` always warned), pinned by `tests/test_sdk_file_mutation_tools.py`. `file_mutation_tool_manifests()` returns the two in stable order (write, edit); `file_mutation_tool_names()` / `is_file_mutation_manifest()` are the matching helpers. The host seam `file_mutation_tool_host(manifest, handler)` / `file_mutation_tool_hosts(handlers)` mirrors `file_tools`'s seam through the non-native `capability_host.BundleHost` (the host does **not** enforce danger — a non-native `BundleHost` accepts a `destructive` `write` bundle exactly as a `safe` `read` bundle; gating is the guard bridge's job). **It migrates no implementation:** it never imports or calls the real `write`/`edit` (which bind to `agent._file_io`/`agent._working_dir`) — handlers are supplied by the wrapper bridge `lingtai.core.file_bundle`; importing `file_mutation_tools` pulls in no `lingtai` wrapper module (asserted by `tests/test_sdk_file_mutation_tools.py::test_file_mutation_tools_import_is_pure_and_migrates_nothing`). The wrapper-side bridge (proving the side-effect bundle-execution pattern against real behavior, including the real overwrite / read-modify-write) is covered by `tests/test_file_bundle_bridge.py`.
- `bundles/lifecycle_tools.py` — **the high-state `system` lifecycle bundle declaration + a native host seam (stage 3C)** (import-pure of the *wrapper*; imports only `.contracts` + `.host` + `.core` + `..errors`): the privileged, native-only mirror of `bundles/file_tools.py` / `bundles/file_mutation_tools.py`, and the migration **template** for the high-state `system` tool. It does **not** redeclare the `system` manifest — `lifecycle_system_manifest()` re-exports `core_bundles.system_bundle()` verbatim (single source of truth: `required`/`privileged`/`native_only`/`NATIVE_ONLY`/`native`, one public `system` tool, bundle-level `danger=destructive`). On top of that one manifest it adds a **per-action risk table** `SYSTEM_ACTION_RISK` (`{action: SecurityDanger}`) plus `KARMA_ACTIONS` / `NIRVANA_ACTIONS` / `SELF_ACTIONS` partitions and an `action_risk(action)` helper (unknown action → conservatively `destructive`): a single bundle-level posture cannot express that `sleep` (self only, no authority) and `nirvana` (irreversible teardown) sit at opposite risk ends, so the strongest action's grade stays the bundle posture and the **graded table ships as a declaration** that mirrors the authority the real kernel intrinsic enforces in code (`lingtai_kernel.intrinsics.system.karma._KARMA_ACTIONS` / `_NIRVANA_ACTIONS`) — never a second gate. The host seam `system_lifecycle_host(handler)` / `system_lifecycle_hosts({"system": handler})` wraps `core_bundles.native_core_host` to host the `system` bundle **alone** (without `psyche`/`soul`, which the all-three `native_core_hosts` requires) as a `capability_host.NativeBundleHost` (native authority); the non-native `BundleHost` refuses it. **It migrates no implementation:** it never imports or calls the real `system` (a *kernel intrinsic* wired live by `BaseAgent._wire_intrinsics`, bound to `agent._admin`/`agent._working_dir`/…) — the handler is supplied by the wrapper bridge `lingtai.core.system_bundle`; importing `lifecycle_tools` pulls in no `lingtai` wrapper module (asserted by `tests/test_sdk_lifecycle_tools.py::test_lifecycle_tools_import_is_pure_and_migrates_no_wrapper`). The danger posture is a **declaration only** — no guard is installed and live dispatch is unchanged; its observable consequence is what the stage-17 `guard_bridge` derives (`system` denied in `BLOCKING` / warned in `ADVISORY`), pinned by `tests/test_sdk_lifecycle_tools.py`. The wrapper-side bridge (hosting the real kernel intrinsic, proving parity incl. the karma/nirvana authority gate) is covered by `tests/test_system_bundle_bridge.py`.
- `bundles/communication_tools.py` — **the high-state `email` / `daemon` communication/execution bundle declarations + host seams (stage 3D)** (import-pure; imports only `.contracts` + `.host` + `..errors`): the communication/execution counterpart of `bundles/lifecycle_tools.py`, declaring the two high-state surfaces with **external or process side effects** — `email` (can send to external SMTP/IMAP recipients) and `daemon` (spawns/kills child processes, runs long child-agent executions). Crucially, the two ride **different live carriers** and the manifests mirror each: `email_comm_manifest()` declares `native` transport + `privileged=True` + `native_only=False` + `backend_replaceability=AUGMENTABLE` (it is a *kernel intrinsic* like `system`, but a backend could re-implement a mail surface) → hosted by `capability_host.NativeBundleHost`; `daemon_exec_manifest()` declares `in_process` + `privileged=False` + `REPLACEABLE` (it is a *wrapper capability* registered via `add_tool`, like the file tools) → hosted by the non-native `capability_host.BundleHost`. Each carries a **per-action risk table** (`EMAIL_ACTION_RISK`: read-only queries `safe`, mailbox/contact mutations + outbound sends `caution`, `delete` `destructive`; `DAEMON_ACTION_RISK`: status queries `safe`, `emanate`/`ask`/`reclaim` `destructive`) plus `email_action_risk(action)` / `daemon_action_risk(action)` helpers (unknown → conservatively `destructive`) and the `EMAIL_SEND_ACTIONS` / `DAEMON_PROCESS_ACTIONS` attention subsets; the bundle-level posture equals the strongest action's grade (`destructive` for both). Host seams: `email_comm_host(handler)` → `NativeBundleHost(native_authority=True)`; `daemon_exec_host(handler)` → `BundleHost`; `communication_tool_hosts({...})` builds both from a `{name: callable}` mapping, refusing a missing/undeclared/extra/non-callable handler. **It migrates no implementation:** it never imports or calls the real `email` (kernel intrinsic) or `daemon` (`DaemonManager.handle`) — handlers are supplied by the wrapper bridge `lingtai.core.communication_bundle`; importing `communication_tools` pulls in no `lingtai` wrapper module (asserted by `tests/test_sdk_communication_tools.py::test_communication_tools_import_is_pure_and_migrates_no_wrapper`). The danger posture is a **declaration only** — no guard is installed and live dispatch is unchanged; its observable consequence is what the stage-17 `guard_bridge` derives (both denied in `BLOCKING` / warned in `ADVISORY`), pinned by `tests/test_sdk_communication_tools.py`. The wrapper-side bridge (hosting the real handlers, proving parity with side-effect-free actions) is covered by `tests/test_communication_bundle_bridge.py`. `mcp` was **deferred** at this stage (its live handler was an inline closure in `setup()` with no stable extractable seam); stage 3F (`bundles/mcp_tools.py`, below) extracts the seam and bridges it.
- `bundles/mcp_tools.py` — **the `mcp` tool-config/catalog bundle declaration + an in-process host seam (stage 3F)** (import-pure; imports only `.contracts` + `.host` + `..errors`): the tool-config/catalog counterpart of `bundles/communication_tools.py`, declaring the single low-side-effect `mcp` surface (the agent's per-agent MCP-server registry view) that stage 3D deferred. Stage 3D excluded `mcp` because its live handler was an inline closure with no extractable seam; stage 3F resolves that — the wrapper now exposes `lingtai.core.mcp.make_handler(agent)` (the same closure `setup()` registers), so the surface has a factory seam exactly like `daemon.make_handler`. `mcp_config_manifest()` declares `in_process` transport + `privileged=False` + `REPLACEABLE` (it is a *wrapper capability* registered via `add_tool`, like `daemon` and the file tools) → hosted by the non-native `capability_host.BundleHost`, mirroring its live carrier. Its single action enum is **`show` only** (a language-neutral copy of `lingtai.core.mcp.get_schema`); the per-action risk table `MCP_ACTION_RISK` grades `show` `safe` and the bundle-level posture equals the strongest action (`safe`). `show` is read-only (re-reads `mcp_registry.jsonl`, re-renders the `mcp` prompt section, returns a health snapshot — no registry write, no MCP server start) but **agent-state-sensitive** (it touches the rendered prompt section, bound to a live `agent`), which is why it rides the in-process capability carrier rather than an out-of-process transport; the manifest metadata records `config`/`catalog`/`read_only`/`agent_state_sensitive`. The `mcp_action_risk(action)` helper fails safe **high** (unknown → `destructive`, the same fail-safe-high direction the other stage-3 action-risk helpers use; system/email/daemon unknown actions also grade `destructive`). Host seams: `mcp_config_host(handler)` → `BundleHost`; `mcp_config_hosts({...})` builds `{name: host}` from a `{name: callable}` mapping, refusing a missing/undeclared/extra/non-callable handler; `mcp_config_manifests()` / `mcp_config_names()` / `is_mcp_config_manifest()` are the matching helpers. **It migrates no implementation:** it never imports or calls the real `mcp` (a wrapper capability whose `make_handler` binds to `agent._working_dir`) — the handler is supplied by the wrapper bridge `lingtai.core.mcp_bundle`; importing `mcp_tools` pulls in no `lingtai` wrapper module (asserted by `tests/test_sdk_mcp_tools.py::test_mcp_tools_import_is_pure_and_migrates_no_wrapper`). The danger posture is a **declaration only** — no guard is installed and live dispatch is unchanged; because `mcp` is `safe`, the stage-17 `guard_bridge` derives a clean pass-through (no deny, no warn) in either mode, pinned by `tests/test_sdk_mcp_tools.py`. The agent's life-state runs on its own loop thread, so it is **sampled** (not pushed): `events()` (`native.py:530`) calls `_sample_agent_state` (`native.py:629`), which reads `agent._state`, **maps it onto the `RuntimeState` taxonomy via `_map_agent_state` (`native.py:102`)** — known wrapper `AgentState` values map to the same-named `RuntimeState`, and any unknown/abnormal value coerces to `STUCK` rather than leaking verbatim (stage 15) — and appends a `STATE` event (payload `data['state']` is always a valid `RuntimeState` value) only when the *mapped* state changed since the last read (so two raw states collapsing to one `RuntimeState` emit once). Event appends and the last-sampled-state read/write are guarded by `self._lock` (the hooks fire on the agent's loop thread). Missing hooks/accessors are tolerated via `getattr` guards. `bridge_events=False` falls back to the stage-1 lifecycle/notification/error snapshot only. **Stage-9 core bundle seam:** every session exposes `core_bundle_manifests` (the stable `system`/`psyche`/`soul` manifests from `bundles/core.py`) as visible native capability contract; if `NativeRuntime(core_handlers=...)` is supplied, `create_session()` validates/builds `core_bundle_hosts` via `native_core_hosts()` as `NativeBundleHost`s around the injected callables only. Omitting handlers leaves the session manifest-only; missing/extra/non-callable handlers raise `BundleHostError`; no real core implementation is imported or called. **Stage-14 lifecycle hardening:** `start()` wraps the boot (LLM-apply + factory + bridge install + `agent.start()`) in a try/except — any raise triggers `_rollback_failed_start` (clears `_agent`, reverses `applied['llm']` back to `deferred`, normalizes state to `PENDING`), emits a **fatal** `ERROR` event, and raises `NativeRuntimeStartError` chaining the original via `__cause__` (generic message, no secret leak); the rolled-back session is restartable. The pre-build `NativeRuntimeConfigurationError` from `_llm_service_from_options` is left to propagate unchanged (session untouched). `send()` wraps the underlying `agent.send(...)` — a raise becomes a **non-fatal** `ERROR` event and `send()` returns (no `NOTIFICATION`), leaving the session `ACTIVE`; the not-active branch is unchanged. `stop()` adds a dirty-join probe: after `agent.stop(timeout)`, if `getattr(agent, "_thread", None)` is still alive (guarded by callable `is_alive()`), it emits a **non-fatal** `ERROR` event; state still becomes `STOPPED`. Covered by `tests/test_sdk_native_runtime_lifecycle.py`.

## Seeds vs. implementations

This package implements only the runtime *adapter* boundary; everything below it stops at the contract. **Present** (stage 1 + stage 2 + stage 3 + stage 4 + stage 9): the `NativeRuntime` skeleton in `bundles/native.py` — it wraps the existing `Agent`, drives its start/stop lifecycle, builds the `LLMService` the default factory needs from `RuntimeOptions` (stage 2), derives that config plus recognized provider defaults / `context_window` from `manifest['llm']` when explicit fields are absent (stage 3), and bridges the running agent's existing activity (tool calls/results, response text, usage, life-state) onto `RuntimeEvent`s via hook-wrapping + state sampling (stage 4), and exposes the core bundle manifest/host seam through injected handlers only (stage 9). It still does not change the kernel turn loop or implement a non-native backend. The following remain **intentionally deferred** to later PRs and are NOT present here:

- A non-native backend (e.g. an Anthropic backend). (init.json-level manifest translation for the default-factory `LLMService` landed in stage 3 — `provider`/`model`/`base_url`/`api_key`, plus `api_compat`/`default_headers`/`compact_threshold`/`max_rpm` via `build_provider_defaults_from_manifest_llm`, and `context_window`. Migrating the privileged `system`/`psyche`/`soul` bundles into `BundleManifest` form is still out of scope.)
- A *streaming/blocking* event bridge — stage 4 added a **minimal** live bridge (tool calls/results, response text, usage, life-state via hook-wrapping + sampling on `events()` reads), but `events()` is still a non-blocking snapshot: there is no incremental token streaming, no async/await or generator that blocks until the next event, and no bridging of intra-turn provider chunks. Those richer surfaces remain deferred.
- **The core `system` / `psyche` / `soul` _implementations_** (not their manifests). The manifest/load/host *boundary* is proven end to end against harmless bundles: the synthetic `proof_bundle()`, the *real, committed* `sdk_skill.sdk_skill_bundle()` (stage 6 — a non-privileged read-only skill asset hosted across tool/resource/prompt surfaces), and the synthetic `native_privileged_proof_bundle()` (stage 7), hosted by `capability_host.NativeBundleHost` to prove the *privileged* native-authority hosting boundary. **As of stage 8** the three core bundles are *declared* as `BundleManifest`s in `bundles/core.py` (each `required`/`privileged`/`native_only`/`NATIVE_ONLY`/`native` with one public tool), and `native_core_host()`/`native_core_hosts()` are the adapter shim that hosts them via `NativeBundleHost` from **injected** handlers. But the **implementations stay deferred**: stage 8 names no real privileged behavior — it never imports or calls the real `system`/`psyche`/`soul`, never moves or rewrites them, and never touches the kernel turn loop. The handler callables are supplied by the native runtime in a *later* stage; this stage only declares the contract and the stub-injection seam around supplied callables. `BundleHost` still refuses every core (privileged/native-only) bundle. Migrating the real privileged behavior into these manifests, and wiring the native runtime's injected handlers, remain later, higher-risk PRs.
- **Live guard wiring.** Stage 17 (`guard/bridge.py`) builds a kernel `GuardCheck` / `ToolCallGuard` from `BundleManifest` posture, but **installs it nowhere**: no `BaseAgent._tool_call_guard` is populated from a bundle set, the wrapper `Agent` / `ToolExecutor` / MCP wiring are untouched, and no behavior-visible runtime change occurs (no C3). Choosing *which* manifests gate a live agent, and installing the resulting guard, is the deferred C3 slice.
- A distribution/package rename (the SDK ships inside the existing `lingtai` wheel for now).

See `docs/sdk/architecture-foundation.md` for the staged roadmap and rationale.

## State

`lingtai_sdk` is **stateless** — it writes nothing to disk and holds no runtime state. Its only side effects are the PEP 562 attribute cache populated in the module's own `globals()` when a lazy wrapper name is first accessed, and **read-only** reads of the committed `assets/` tree (`sdk_skill.load_sdk_skill()` reads `SKILL.md` via `importlib.resources`; it never writes).

## Notes

- Adding a public name: prefer re-exporting from `types.py`/`errors.py` (kernel-backed, eager) or `_LAZY_WRAPPER_EXPORTS` (wrapper-backed, lazy). Update `__all__` and, if it formalizes a legacy path, add a `_compat.Deprecation` entry.
- Do not import heavy provider SDKs at module load anywhere in this package; the import-purity tests will fail. Bare `google` (an ambient namespace stub pulled in transitively by `filelock`) is harmless and explicitly excluded from the purity check — only heavy submodules like `google.genai` count.
- Adding an SDK asset: drop it under `assets/<skill-name>/` (a skill-shaped layout), read it via `importlib.resources` (never a raw filesystem path), and rely on the `lingtai_sdk = ["assets/**/*"]` `package-data` glob so it ships in wheels. Keep assets read-only and import-pure. Update `assets/ANATOMY.md`.
