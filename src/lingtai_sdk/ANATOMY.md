# lingtai_sdk

> **Maintenance:** see the `lingtai-kernel-anatomy` skill. **Coding agents** update this file in the same commit as code changes. **LingTai agents** report drift as issues (mail or `discussions/<name>-patch.md`); do not silently fix.

The **public SDK doorway** — a curated, import-light front door for building and embedding LingTai agents. It re-exports the two implementation packages (`lingtai_kernel`, `lingtai`) under one stable path and carries the *seed contracts* (runtime, capability-bundle manifest) that later PRs will implement. This is a foundation package: it ships shapes and a doorway, not a live runtime.

> **What is an `ANATOMY.md`?** See the canonical convention at `src/lingtai/intrinsic_skills/lingtai-kernel-anatomy/SKILL.md`. This file follows the same 6-section template as every other anatomy in the tree.

## Components

- `__init__.py` — the curated public surface. Eager kernel re-exports (`BaseAgent`, plus `types`/`errors` names); wrapper-backed names (`Agent`, `FileIOService`, `MailService`, `LoggingService`, `SearchService`, `VisionService`) resolve **lazily** via :pep:`562` `__getattr__` against `_LAZY_WRAPPER_EXPORTS`, caching into `globals()` on first access. `__all__` is the canonical export list; `__dir__` mirrors it. Stage-10/11/13 SDK-internal lazy exports also expose `LingTaiClient`, `LingTaiSession`, `QueryResult`, `open_session`, `query`, and the runtime-contract DTOs/ABCs from `runtime.py` without importing the wrapper.
- `_version.py` — `__version__`, resolved best-effort from the `lingtai` distribution metadata (`importlib.metadata.version("lingtai")`), falling back to `"0+unknown"` so import never fails over missing metadata.
- `types.py` — re-exports kernel public types (`AgentConfig`, `AgentState`, `Message`, `MSG_REQUEST`, `MSG_USER_INPUT`, `ChatSession`, `FunctionSchema`, `LLMResponse`, `ToolCall`, `LLMService`) under a stable path. Kernel-only; no heavy provider SDK is pulled.
- `errors.py` — `LingTaiSDKError` (SDK base), `NativeRuntimeConfigurationError` (raised by the default native factory when LLM config is partial/absent, *before* any agent is built), `NativeRuntimeStartError` (stage 14 — raised when `start()` fails *during* boot: factory/construction/`agent.start()` raised; the session rolls back to safe `PENDING`/no-agent first; the original is chained via `__cause__` and the message never echoes secrets), plus a re-export of kernel `UnknownToolError`. Import-pure (kernel `types` only).
- `_compat.py` — the machine-readable migration map: `Deprecation` dataclass, `DEPRECATIONS` tuple, `active_aliases()`, `migration_for()`. Drives the docs migration table and the same-object round-trip test.
- `client.py` — **thin public client/session facade** (stage 10 + stage 11 + stage 12 polish + stage 15 event-semantics fix, import-pure): `LingTaiClient` wraps any supplied `Runtime` and offers `query(message, ...)` plus `open_session()`; `LingTaiSession` is a small wrapper over a live `RuntimeSession` for multi-message use (`send` / `events` / `text` / `close`, context-manager close); `text()` is a text-only convenience drain that discards non-text events from its returned string, so callers needing state/tool/usage/error/raw data should call `events()` directly; `QueryResult` carries concatenated `TEXT` event output plus the full `RuntimeEvent` tuple; module-level `query()` is a one-shot helper and `open_session()` starts a live session helper. **Stage-15 event semantics:** because `RuntimeSession.events()` is a *non-draining cumulative snapshot* (see `runtime.py`), `query()` now reads it **exactly once** after the lifecycle settles (`native.py`/`client.py`) — it no longer extends with `events()` after `send` *and* again after `stop`, which double-counted the early events against a real (snapshot) session (the old fake masked it by draining on read). `LingTaiSession` keeps its **own read cursor** so `events()`/`text()` return only the events appended since the previous read (an incremental drain-like view) without mutating the session; `raw_session` still exposes the full snapshot. It owns no backend behavior: tests inject a fake runtime (now contract-conformant/non-draining), and the default native runtime is imported lazily only when no runtime is supplied. No wrapper `Agent`, provider SDK, Anthropic backend, or kernel turn-loop import/change occurs here.
- `runtime.py` — **runtime contract seed** (pure DTOs/ABCs, no kernel import): `RuntimeState`, `EventKind`, `RuntimeOptions`, `RuntimeMessage`, `RuntimeEvent` (with `state`/`text`/`error` constructors plus `tool_call`/`tool_result`/`usage` activity constructors — `runtime.py:110-128`), and the `RuntimeSession`/`Runtime` ABCs. Describes how a *future* live runtime is driven (options in, messages in, events out). **Stage-15 `events()` contract:** the `RuntimeSession` docstring now pins `events()` as a *non-draining, re-iterable cumulative snapshot* — every call returns **all** events so far, in order, and reading never consumes them; backends MUST NOT drain on read (a draining backend breaks `query()`), and incremental "only what's new" views are the caller's responsibility (cursor in `LingTaiSession`). `STATE` event `data['state']` is documented as a `RuntimeState` *value*, so backends map their own life-state onto the taxonomy. No live runtime here.
- `capabilities.py` — **CapabilityBundle manifest seed** (pure DTOs, no `lingtai`/`lingtai_kernel` import): `BackendReplaceability`, **`SecurityDanger`** (`safe`/`caution`/`destructive`) and **`TransportKind`** (`native`/`in_process`/`stdio`/`http`) — public `str`-valued Enums acting as allow-lists for `SecurityPolicy.danger` / `TransportSpec.kind` (the fields stay plain strings, no breaking type change), `RoleFlags`, `CapabilitySurfaces`, `SecurityPolicy`, `TransportSpec`, `BundleManifest` (with `validate()`/`to_dict()`), `load_manifest()` — the inverse of `to_dict()`, parsing a plain-dict declaration back into a *validated* `BundleManifest` (reconstructs the nested frozen dataclasses + enum, raises `BundleLoadError` on unknown enum value / non-mapping block / failed invariant), and `proof_bundle()` — a harmless metadata-only synthetic bundle. Public schema + (de)serialization only; native privileged handlers stay in the kernel/wrapper. Imports only the import-pure `.errors` (for `BundleLoadError`). **Stage-7 strict validation:** `validate()` now also rejects an unknown `security.danger` / `transport.kind` (so a *directly constructed* manifest is checked too); `load_manifest()` reads role flags (`required`/`privileged`/`native_only`/`can_override`) as *real* booleans via `_strict_bool` (a present non-`bool` like `"false"`/`1` is rejected — `bool` checked before `int` since `True`/`False` are `int`s — an absent flag falls back to its default), and coerces `metadata` / `transport.config` via `_mapping_block` (a non-mapping list/scalar is rejected with a clear `BundleLoadError`, never half-coerced).
- `capability_host.py` — **CapabilityBundle host boundary** (import-pure; imports only `.capabilities` + `.errors`): a private `_BaseBundleHost` holds the shared contract — validate the manifest on registration, enforce the manifest↔handler contract per surface via `_check_contract` (every declared `surfaces.tools`/`surfaces.resources`/`surfaces.prompts` name has a callable handler, no handler is undeclared), store handlers, and expose three read-only surfaces (`invoke(tool, **kwargs)` / `read_resource(name)` / `read_prompt(name, **kwargs)`, each raising `BundleHostError` for an undeclared name). Subclasses differ only in **admission** (`_admit(manifest)`, called after `validate()`, before any handler is stored): **`BundleHost`** is the *non-native* host — it **refuses** `privileged`/`native_only` bundles and non-`in_process` transports (the unchanged guardrail that keeps `system`/`psyche`/`soul` out); **`NativeBundleHost`** (stage 7) is the *native-authority* host — it may host a `privileged`/`native_only` bundle, but **only** when explicitly constructed with the keyword-only `native_authority=True` (defaults `False`, so privileged hosting is never granted by accident) and **only** for a `native` transport, applying the *same* validation and per-surface contract. `proof_host()` wires `proof_bundle()` to a deterministic `_echo` handler; `native_privileged_proof_bundle()` + `native_proof_host()` are the privileged-side mirror — a harmless synthetic privileged/`native_only`/`native`-transport bundle with one deterministic `native_noop` tool, hosted only because the host carries explicit authority. No real privileged surface is named; the privileged `system`/`psyche`/`soul` bundles are deliberately NOT migrated here (stage 8).
- `core_bundles.py` — **the `system` / `psyche` / `soul` core bundle manifests + a native adapter shim (stage 8)** (import-pure; imports only `.capabilities` + `.capability_host` + `.errors`): the first deliberate contact with the privileged core surfaces, **as manifest contract + stub-injection seam only**. `system_bundle()` / `psyche_bundle()` / `soul_bundle()` each return a `BundleManifest` built by the shared `_core_manifest` helper with the identical privileged posture — `required=True`, `privileged=True`, `native_only=True`, `backend_replaceability=NATIVE_ONLY`, `transport.kind == native` — declaring exactly one public tool named after the bundle (`system`/`psyche`/`soul`) and helpful, **non-secret** metadata (`core`/`role`/`actions`). They differ only in declared `SecurityDanger`: `system` is `destructive` (lifecycle control incl. irreversible teardown), `psyche` is `caution` (identity + context shedding has lasting effects), `soul` is `caution` (inner-voice consultation plus persisted flow/voice config). `core_bundle_manifests()` returns the three in stable order (system, psyche, soul); `core_bundle_names()` / `is_core_manifest()` are the matching helpers. The adapter shim `native_core_host(manifest, handler)` turns a core manifest plus a single **injected** callable into a `capability_host.NativeBundleHost` constructed with `native_authority=True` (refusing a non-core manifest or a non-callable handler with `BundleHostError`, then deferring manifest/handler-parity + native-authority/transport enforcement to `NativeBundleHost`); `native_core_hosts(handlers)` builds `{name: NativeBundleHost}` for all three from a `{name: callable}` mapping, refusing a missing or undeclared/extra handler (so a partial wiring can never boot a subset of the privileged core silently). **It migrates no implementation:** it never imports or calls the real `system`/`psyche`/`soul` — handlers are supplied by the native runtime in a later stage; importing `core_bundles` pulls in no `lingtai` wrapper module (asserted by `tests/test_sdk_core_bundles.py::test_core_bundles_import_is_pure_and_migrates_nothing`).
- `file_tools.py` — **the low-state `read` / `glob` / `grep` file-tool bundle manifests + an in-process host seam (stage 3A)** (import-pure; imports only `.capabilities` + `.capability_host` + `.errors`): the non-privileged, in-process mirror of `core_bundles.py` and the migration **template** for low-state file/query tools. `read_bundle()` / `glob_bundle()` / `grep_bundle()` each return a `BundleManifest` built by the shared `_file_tool_manifest` helper with the identical low-state posture — `required=False`, `privileged=False`, `native_only=False`, `backend_replaceability=REPLACEABLE`, `transport.kind == in_process` — declaring exactly one public tool named after the bundle (`read`/`glob`/`grep`) and helpful, **non-secret** metadata (`file_tool`/`role`/`actions` + a copy of the tool's argument `schema`, a language-neutral mirror of `lingtai.core.{read,glob,grep}.get_schema`). All three are read-only / query surfaces, so all declare `SecurityDanger.SAFE`. `file_tool_manifests()` returns the three in stable order (read, glob, grep); `file_tool_names()` / `is_file_tool_manifest()` are the matching helpers. The host seam `file_tool_host(manifest, handler)` turns a file-tool manifest plus a single **injected** callable into a `capability_host.BundleHost` (refusing a non-file-tool manifest or a non-callable handler with `BundleHostError`, then deferring manifest/handler-parity + non-privileged/`in_process` enforcement to `BundleHost`); `file_tool_hosts(handlers)` builds `{name: BundleHost}` for all three from a `{name: callable}` mapping, refusing a missing or undeclared/extra handler. **It migrates no implementation:** it never imports or calls the real `read`/`glob`/`grep` (which bind to the wrapper's `agent._file_io`/`agent._working_dir`) — handlers are supplied by the wrapper bridge `lingtai.core.file_bundle`; importing `file_tools` pulls in no `lingtai` wrapper module (asserted by `tests/test_sdk_file_tools.py::test_file_tools_import_is_pure_and_migrates_nothing`). Unlike the privileged core bundles, the non-native `BundleHost` *accepts* these directly. The wrapper-side bridge (which proves the bundle-execution pattern against the real behavior) is covered by `tests/test_file_bundle_bridge.py`.
- `guard_bridge.py` — **the pure `BundleManifest` → kernel `GuardCheck` bridge (stage 17)** (import-pure; imports only kernel `lingtai_kernel.tool_call_guard` + the import-pure `.capabilities`): turns one or more `BundleManifest`s into a kernel `GuardCheck` callable / a ready `ToolCallGuard` chain so the stage-16 `ToolExecutor` guard seam can consult declared bundle posture **before** dispatch — *without* any executor / `BaseAgent` / wrapper `Agent` change (no live wiring; that is a later C3 stage). `tool_danger_index(manifests)` builds `{tool_name: SecurityDanger}` (most-dangerous posture wins on a duplicate tool name; an unrecognized danger string is treated conservatively as `destructive`); `guard_check_from_manifests(manifests, *, mode=BLOCKING)` returns the `GuardCheck`, `tool_call_guard_from_manifests(...)` the one-check `ToolCallGuard`. **Policy** (narrow/deterministic/**fail-open**): `safe` → allow clean pass-through (check returns `None`, no advisory); `caution` → allow-but-`warn` (advisory carries `bundle`/`danger`/`policy_mode`); `destructive` → `GuardPolicyMode.BLOCKING` (default) **denies**, `GuardPolicyMode.ADVISORY` allows-with-`warn`; an **unknown** tool (declared by no supplied manifest) → **allow** (never fails closed — installing the bridge can only add denials for explicitly-declared destructive surfaces). Dependency is one-way: the SDK imports kernel guard types, the kernel never imports the SDK (asserted by `tests/test_sdk_guard_bridge.py`). Imports **no** `lingtai` wrapper / provider SDK.
- `sdk_skill.py` — **the committed `lingtai-sdk-skill` asset + its bundle (stage 6)** (import-pure; imports only `importlib.resources` + `.capabilities` + `.capability_host`): the first *real, low-risk* CapabilityBundle adoption (not synthetic echo). `load_sdk_skill()` reads the committed `SKILL.md` under `assets/lingtai-sdk-skill/` via `importlib.resources` (source checkout *or* installed wheel). `sdk_skill_bundle()` declares a non-privileged, replaceable, `in_process` `BundleManifest` over it — one read-only tool (`read_sdk_skill`), one resource (`sdk_skill`), one prompt (`sdk_skill_orientation`), with `manual=(SKILL_MANUAL_PATH,)`. `sdk_skill_host()` returns a ready `BundleHost` wiring those three surfaces to deterministic, network-free handlers reading the asset — the end-to-end asset→`load_sdk_skill`→`sdk_skill_bundle`→`BundleHost`→`read_resource`/`invoke`/`read_prompt` adoption. No privileged behavior; `system`/`psyche`/`soul` are NOT migrated.
- `assets/` — committed, read-only **resource package** (`assets/__init__.py` is a marker only; no code/runtime). Holds the shipped `lingtai-sdk-skill/SKILL.md` — the top-level SDK *observation entry* — reachable as the `lingtai_sdk.assets` resource root. See `assets/ANATOMY.md`. Packaged via the `lingtai_sdk = ["assets/**/*"]` `package-data` glob in `pyproject.toml`.
- `native.py` — **live `NativeRuntime` adapter skeleton** (NOT a full backend; stage 1 + stage-2 LLM-service translation + stage-3 manifest translation + stage-9 core bundle seam). `NativeRuntime(Runtime)` (id `"native"`) is a factory for `NativeRuntimeSession(RuntimeSession)`, which wraps the wrapper `Agent` unchanged: translates `RuntimeOptions` → `Agent` kwargs (`_agent_kwargs_from_options`), drives `start()`/`stop()` lifecycle (`PENDING`→`ACTIVE`→`STOPPED`), routes `send()` onto `Agent.send()` (fire-and-forget queue), and exposes a non-blocking `events()` snapshot of lifecycle/notification/error events. Import-pure at module load (imports only `runtime` + the import-pure `errors`); the wrapper `Agent` and `lingtai.llm` service/builder are imported lazily on first `start()`. **Stage-2 LLM-service translation:** on the *default* factory path (`agent_factory` not injected), `start()` builds the `LLMService` the wrapper `Agent` requires via `_llm_service_from_options(options)` (lazy `lingtai.llm` import), passes it as `service`, and moves the secret-free LLM config from `session.deferred['llm']` into `session.applied['llm']` (`_public_llm_fields` strips `api_key`). Partial/absent `provider`/`model` raises `NativeRuntimeConfigurationError` *before* any agent is built (session stays `PENDING`) — the opaque missing-`service` `TypeError` no longer leaks (GLM stage-1 nit N2). An injected `agent_factory` **bypasses** service building (`_uses_default_factory` flag), so fakes/hosts boot without a real service. `api_key` is consumed into the service and never stored on the session, surfaced in events, or echoed in errors/reprs. **Stage-3 manifest translation (pure helpers):** `_llm_config_from_options` merges explicit `provider`/`model`/`base_url`/`api_key` over `manifest['llm']` (explicit wins, manifest fills gaps); `_max_rpm_from_options_or_manifest` resolves `max_rpm` (`extra['native']` → `extra` → `manifest` → `manifest['llm']`, default `0` = no SDK-imposed gating); `_context_window_from_options_or_manifest` resolves `context_window` (`manifest['llm']` → `manifest['context_limit']` → `extra['native']` → `extra`, else `None`). `_llm_service_from_options` builds from the merged config and, when a `manifest['llm']` block is present, plumbs recognized provider defaults (`api_compat`, `default_headers`, `compact_threshold`, `max_rpm`) through `build_provider_defaults_from_manifest_llm` **scoped to the merged provider**, plus `context_window`. `_sanitized_manifest` redacts `manifest['llm']['api_key']` from `session.deferred['manifest']` (the original `RuntimeOptions.manifest` is untouched). The merge runs before the provider/model presence check, so manifest-only config builds and partial config still raises a secret-free `NativeRuntimeConfigurationError`. **Stage-4 live event bridge:** when `bridge_events` is `True` (the default, threaded through `NativeRuntime` → `NativeRuntimeSession`), `start()` calls `_install_event_bridge(agent)` (`native.py:460`) right after the agent is built. The bridge wraps the agent's overridable hooks as *instance* attributes — purely additive, no kernel turn-loop change: `_on_tool_result_hook` → `TOOL_CALL` + `TOOL_RESULT` events (the original's return value passes through, so an intercepting host hook still short-circuits the turn); `_post_request` → a `TEXT` event for non-empty `result['text']`, a non-fatal `ERROR` per `result['errors']`, and a `USAGE` event sampled from `agent.get_token_usage()` (`_emit_turn_result` — `native.py:610`). The agent's life-state runs on its own loop thread, so it is **sampled** (not pushed): `events()` (`native.py:530`) calls `_sample_agent_state` (`native.py:629`), which reads `agent._state`, **maps it onto the `RuntimeState` taxonomy via `_map_agent_state` (`native.py:102`)** — known wrapper `AgentState` values map to the same-named `RuntimeState`, and any unknown/abnormal value coerces to `STUCK` rather than leaking verbatim (stage 15) — and appends a `STATE` event (payload `data['state']` is always a valid `RuntimeState` value) only when the *mapped* state changed since the last read (so two raw states collapsing to one `RuntimeState` emit once). Event appends and the last-sampled-state read/write are guarded by `self._lock` (the hooks fire on the agent's loop thread). Missing hooks/accessors are tolerated via `getattr` guards. `bridge_events=False` falls back to the stage-1 lifecycle/notification/error snapshot only. **Stage-9 core bundle seam:** every session exposes `core_bundle_manifests` (the stable `system`/`psyche`/`soul` manifests from `core_bundles.py`) as visible native capability contract; if `NativeRuntime(core_handlers=...)` is supplied, `create_session()` validates/builds `core_bundle_hosts` via `native_core_hosts()` as `NativeBundleHost`s around the injected callables only. Omitting handlers leaves the session manifest-only; missing/extra/non-callable handlers raise `BundleHostError`; no real core implementation is imported or called. **Stage-14 lifecycle hardening:** `start()` wraps the boot (LLM-apply + factory + bridge install + `agent.start()`) in a try/except — any raise triggers `_rollback_failed_start` (clears `_agent`, reverses `applied['llm']` back to `deferred`, normalizes state to `PENDING`), emits a **fatal** `ERROR` event, and raises `NativeRuntimeStartError` chaining the original via `__cause__` (generic message, no secret leak); the rolled-back session is restartable. The pre-build `NativeRuntimeConfigurationError` from `_llm_service_from_options` is left to propagate unchanged (session untouched). `send()` wraps the underlying `agent.send(...)` — a raise becomes a **non-fatal** `ERROR` event and `send()` returns (no `NOTIFICATION`), leaving the session `ACTIVE`; the not-active branch is unchanged. `stop()` adds a dirty-join probe: after `agent.stop(timeout)`, if `getattr(agent, "_thread", None)` is still alive (guarded by callable `is_alive()`), it emits a **non-fatal** `ERROR` event; state still becomes `STOPPED`. Covered by `tests/test_sdk_native_runtime_lifecycle.py`.

## Connections

- **The kernel must never import this package.** Dependency flows one way: `lingtai_sdk` → (`lingtai`, `lingtai_kernel`). The SDK is a consumer of the other two, never a dependency of them.
- **Eager kernel, lazy wrapper.** `import lingtai_sdk` loads the dependency-light kernel only. Wrapper-backed names (`_LAZY_WRAPPER_EXPORTS`) import `lingtai` on first attribute access, keeping the wrapper's provider SDKs (anthropic/openai/google-genai/mcp/…) optional for pure-kernel consumers. Enforced by `tests/test_sdk_import_purity.py`.
- **Lazy SDK-internal names.** `NativeRuntime`/`NativeRuntimeSession` (`_LAZY_SDK_EXPORTS`) resolve from the import-pure `.native` module, NOT from the wrapper — accessing them and constructing a `NativeRuntime` stays provider-free; the wrapper `Agent` loads only when a session is started. Stage 13 extends the same lazy table to the import-pure runtime contract names (`RuntimeOptions`, `RuntimeMessage`, `RuntimeEvent`, `RuntimeState`, `EventKind`, `Runtime`, `RuntimeSession`) so root-level client helpers can be used without separately importing `lingtai_sdk.runtime`. Enforced by `tests/test_sdk_native_runtime.py::test_importing_native_and_constructing_runtime_is_pure` and `tests/test_sdk_client_facade.py::test_root_exports_runtime_contract_lazily_and_wrapper_free`.
- `runtime.py`, `capabilities.py`, `capability_host.py`, `sdk_skill.py`, `core_bundles.py`, `native.py`, and `client.py` import **nothing from the `lingtai` wrapper** at module load (the purity tests forbid `lingtai`/`lingtai.`, not the dependency-light `lingtai_kernel`). `runtime.py` imports nothing from either; `capabilities.py` and `capability_host.py` import only the import-pure `.errors` (which re-exports `lingtai_kernel.types.UnknownToolError` — kernel, not wrapper); `capability_host.py` also imports `.capabilities`; `sdk_skill.py` imports only `importlib.resources` + `.capabilities` + `.capability_host` and reads its asset through `importlib.resources` (no wrapper, no filesystem-path guessing); `core_bundles.py` imports only `.capabilities` + `.capability_host` + `.errors` (so declaring the core manifests does NOT import the real `system`/`psyche`/`soul`); `file_tools.py` imports only `.capabilities` + `.capability_host` + `.errors` (so declaring the file-tool manifests does NOT import the real `read`/`glob`/`grep`; the wrapper bridge `lingtai.core.file_bundle` injects the real handlers in the *wrapper*, preserving the one-way `wrapper → sdk` direction); `guard_bridge.py` imports only the dependency-light kernel `lingtai_kernel.tool_call_guard` + `.capabilities` — the **one** module that imports a kernel symbol beyond `types`/`errors`, and it is still wrapper-free (the kernel `tool_call_guard` is itself import-light); `native.py` imports the local `runtime` contract plus import-pure `.capabilities` / `.capability_host` / `.core_bundles` / `.errors`, deferring its wrapper `Agent` import to `start()`. Enforced by `tests/test_sdk_runtime_contract.py`, `tests/test_sdk_capabilities.py`, `tests/test_sdk_capability_host.py`, `tests/test_sdk_skill_bundle.py`, `tests/test_sdk_core_bundles.py`, `tests/test_sdk_guard_bridge.py`, `tests/test_sdk_native_runtime.py`, and `tests/test_sdk_client_facade.py`.
- **No import inversion (kernel → SDK).** `guard_bridge.py` imports kernel `tool_call_guard` types, but the kernel `tool_call_guard` module must stand alone — importing it pulls in **no** `lingtai_sdk` (asserted by `tests/test_sdk_guard_bridge.py::test_kernel_does_not_import_sdk`). The bridge is a pure consumer of the kernel guard contract; it adds no live wiring (the stage-16 `ToolExecutor`/`BaseAgent` guard seam is untouched, deferred to a later C3 stage).
- Compatibility is **by re-export, not re-implementation**: `_compat.DEPRECATIONS` records legacy→SDK path moves; the round-trip test asserts each pair resolves to the *same object* (`tests/test_sdk_compat.py`).

## Seeds vs. implementations

This package implements only the runtime *adapter* boundary; everything below it stops at the contract. **Present** (stage 1 + stage 2 + stage 3 + stage 4 + stage 9): the `NativeRuntime` skeleton in `native.py` — it wraps the existing `Agent`, drives its start/stop lifecycle, builds the `LLMService` the default factory needs from `RuntimeOptions` (stage 2), derives that config plus recognized provider defaults / `context_window` from `manifest['llm']` when explicit fields are absent (stage 3), and bridges the running agent's existing activity (tool calls/results, response text, usage, life-state) onto `RuntimeEvent`s via hook-wrapping + state sampling (stage 4), and exposes the core bundle manifest/host seam through injected handlers only (stage 9). It still does not change the kernel turn loop or implement a non-native backend. The following remain **intentionally deferred** to later PRs and are NOT present here:

- A non-native backend (e.g. an Anthropic backend). (init.json-level manifest translation for the default-factory `LLMService` landed in stage 3 — `provider`/`model`/`base_url`/`api_key`, plus `api_compat`/`default_headers`/`compact_threshold`/`max_rpm` via `build_provider_defaults_from_manifest_llm`, and `context_window`. Migrating the privileged `system`/`psyche`/`soul` bundles into `BundleManifest` form is still out of scope.)
- A *streaming/blocking* event bridge — stage 4 added a **minimal** live bridge (tool calls/results, response text, usage, life-state via hook-wrapping + sampling on `events()` reads), but `events()` is still a non-blocking snapshot: there is no incremental token streaming, no async/await or generator that blocks until the next event, and no bridging of intra-turn provider chunks. Those richer surfaces remain deferred.
- **The core `system` / `psyche` / `soul` _implementations_** (not their manifests). The manifest/load/host *boundary* is proven end to end against harmless bundles: the synthetic `proof_bundle()`, the *real, committed* `sdk_skill.sdk_skill_bundle()` (stage 6 — a non-privileged read-only skill asset hosted across tool/resource/prompt surfaces), and the synthetic `native_privileged_proof_bundle()` (stage 7), hosted by `capability_host.NativeBundleHost` to prove the *privileged* native-authority hosting boundary. **As of stage 8** the three core bundles are *declared* as `BundleManifest`s in `core_bundles.py` (each `required`/`privileged`/`native_only`/`NATIVE_ONLY`/`native` with one public tool), and `native_core_host()`/`native_core_hosts()` are the adapter shim that hosts them via `NativeBundleHost` from **injected** handlers. But the **implementations stay deferred**: stage 8 names no real privileged behavior — it never imports or calls the real `system`/`psyche`/`soul`, never moves or rewrites them, and never touches the kernel turn loop. The handler callables are supplied by the native runtime in a *later* stage; this stage only declares the contract and the stub-injection seam around supplied callables. `BundleHost` still refuses every core (privileged/native-only) bundle. Migrating the real privileged behavior into these manifests, and wiring the native runtime's injected handlers, remain later, higher-risk PRs.
- **Live guard wiring.** Stage 17 (`guard_bridge.py`) builds a kernel `GuardCheck` / `ToolCallGuard` from `BundleManifest` posture, but **installs it nowhere**: no `BaseAgent._tool_call_guard` is populated from a bundle set, the wrapper `Agent` / `ToolExecutor` / MCP wiring are untouched, and no behavior-visible runtime change occurs (no C3). Choosing *which* manifests gate a live agent, and installing the resulting guard, is the deferred C3 slice.
- A distribution/package rename (the SDK ships inside the existing `lingtai` wheel for now).

See `docs/sdk/architecture-foundation.md` for the staged roadmap and rationale.

## State

`lingtai_sdk` is **stateless** — it writes nothing to disk and holds no runtime state. Its only side effects are the PEP 562 attribute cache populated in the module's own `globals()` when a lazy wrapper name is first accessed, and **read-only** reads of the committed `assets/` tree (`sdk_skill.load_sdk_skill()` reads `SKILL.md` via `importlib.resources`; it never writes).

## Notes

- Adding a public name: prefer re-exporting from `types.py`/`errors.py` (kernel-backed, eager) or `_LAZY_WRAPPER_EXPORTS` (wrapper-backed, lazy). Update `__all__` and, if it formalizes a legacy path, add a `_compat.Deprecation` entry.
- Do not import heavy provider SDKs at module load anywhere in this package; the import-purity tests will fail. Bare `google` (an ambient namespace stub pulled in transitively by `filelock`) is harmless and explicitly excluded from the purity check — only heavy submodules like `google.genai` count.
- Adding an SDK asset: drop it under `assets/<skill-name>/` (a skill-shaped layout), read it via `importlib.resources` (never a raw filesystem path), and rely on the `lingtai_sdk = ["assets/**/*"]` `package-data` glob so it ships in wheels. Keep assets read-only and import-pure. Update `assets/ANATOMY.md`.
