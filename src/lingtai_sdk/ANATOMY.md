# lingtai_sdk

> **Maintenance:** see the `lingtai-kernel-anatomy` skill. **Coding agents** update this file in the same commit as code changes. **LingTai agents** report drift as issues (mail or `discussions/<name>-patch.md`); do not silently fix.

The **public SDK doorway** — a curated, import-light front door for building and embedding LingTai agents. It re-exports the two implementation packages (`lingtai_kernel`, `lingtai`) under one stable path and carries the *seed contracts* (runtime, capability-bundle manifest) that later PRs will implement. This is a foundation package: it ships shapes and a doorway, not a live runtime.

> **What is an `ANATOMY.md`?** See the canonical convention at `src/lingtai/intrinsic_skills/lingtai-kernel-anatomy/SKILL.md`. This file follows the same 6-section template as every other anatomy in the tree.

## Components

- `__init__.py` — the curated public surface. Eager kernel re-exports (`BaseAgent`, plus `types`/`errors` names); wrapper-backed names (`Agent`, `FileIOService`, `MailService`, `LoggingService`, `SearchService`, `VisionService`) resolve **lazily** via :pep:`562` `__getattr__` against `_LAZY_WRAPPER_EXPORTS`, caching into `globals()` on first access. `__all__` is the canonical export list; `__dir__` mirrors it.
- `_version.py` — `__version__`, resolved best-effort from the `lingtai` distribution metadata (`importlib.metadata.version("lingtai")`), falling back to `"0+unknown"` so import never fails over missing metadata.
- `types.py` — re-exports kernel public types (`AgentConfig`, `AgentState`, `Message`, `MSG_REQUEST`, `MSG_USER_INPUT`, `ChatSession`, `FunctionSchema`, `LLMResponse`, `ToolCall`, `LLMService`) under a stable path. Kernel-only; no heavy provider SDK is pulled.
- `errors.py` — `LingTaiSDKError` (SDK base), `NativeRuntimeConfigurationError` (raised by the default native factory when LLM config is partial/absent), plus a re-export of kernel `UnknownToolError`. Import-pure (kernel `types` only).
- `_compat.py` — the machine-readable migration map: `Deprecation` dataclass, `DEPRECATIONS` tuple, `active_aliases()`, `migration_for()`. Drives the docs migration table and the same-object round-trip test.
- `runtime.py` — **runtime contract seed** (pure DTOs/ABCs, no kernel import): `RuntimeState`, `EventKind`, `RuntimeOptions`, `RuntimeMessage`, `RuntimeEvent` (with `state`/`text`/`error` constructors plus `tool_call`/`tool_result`/`usage` activity constructors — `runtime.py:110-128`), and the `RuntimeSession`/`Runtime` ABCs. Describes how a *future* live runtime is driven (options in, messages in, events out). No live runtime here.
- `capabilities.py` — **CapabilityBundle manifest seed** (pure DTOs, no `lingtai`/`lingtai_kernel` import): `BackendReplaceability`, `RoleFlags`, `CapabilitySurfaces`, `SecurityPolicy`, `TransportSpec`, `BundleManifest` (with `validate()`/`to_dict()`), `load_manifest()` (`capabilities.py:139`) — the inverse of `to_dict()`, parsing a plain-dict declaration back into a *validated* `BundleManifest` (reconstructs the nested frozen dataclasses + enum, raises `BundleLoadError` on unknown enum value / non-mapping block / failed invariant), and `proof_bundle()` (`capabilities.py:226`) — a harmless metadata-only synthetic bundle. Public schema + (de)serialization only; native privileged handlers stay in the kernel/wrapper. Imports only the import-pure `.errors` (for `BundleLoadError`).
- `capability_host.py` — **CapabilityBundle host boundary** (import-pure; imports only `.capabilities` + `.errors`): `BundleHost` (`capability_host.py:66`) hosts a single *validated, non-privileged* bundle in-process across three read-only surfaces — its `__init__` validates the manifest, **refuses** `privileged`/`native_only` bundles (only the native runtime may host those) and non-`in_process` transports, and enforces the manifest↔handler contract per surface via `_check_contract` (every declared `surfaces.tools`/`surfaces.resources`/`surfaces.prompts` name has a callable handler, no handler is undeclared), raising `BundleHostError` on any breach. `invoke(tool, **kwargs)` dispatches a declared tool, `read_resource(name)` reads a declared resource (`name → callable()`), `read_prompt(name, **kwargs)` renders a declared prompt — each raises `BundleHostError` for an undeclared name. `proof_host()` wires `proof_bundle()` to a deterministic, network-free `_echo` handler (tools only). This is the *non-native* host; the privileged `system`/`psyche`/`soul` bundles are deliberately NOT hosted here.
- `sdk_skill.py` — **the committed `lingtai-sdk-skill` asset + its bundle (stage 6)** (import-pure; imports only `importlib.resources` + `.capabilities` + `.capability_host`): the first *real, low-risk* CapabilityBundle adoption (not synthetic echo). `load_sdk_skill()` reads the committed `SKILL.md` under `assets/lingtai-sdk-skill/` via `importlib.resources` (source checkout *or* installed wheel). `sdk_skill_bundle()` declares a non-privileged, replaceable, `in_process` `BundleManifest` over it — one read-only tool (`read_sdk_skill`), one resource (`sdk_skill`), one prompt (`sdk_skill_orientation`), with `manual=(SKILL_MANUAL_PATH,)`. `sdk_skill_host()` returns a ready `BundleHost` wiring those three surfaces to deterministic, network-free handlers reading the asset — the end-to-end asset→`load_sdk_skill`→`sdk_skill_bundle`→`BundleHost`→`read_resource`/`invoke`/`read_prompt` adoption. No privileged behavior; `system`/`psyche`/`soul` are NOT migrated.
- `assets/` — committed, read-only **resource package** (`assets/__init__.py` is a marker only; no code/runtime). Holds the shipped `lingtai-sdk-skill/SKILL.md` — the top-level SDK *observation entry* — reachable as the `lingtai_sdk.assets` resource root. See `assets/ANATOMY.md`. Packaged via the `lingtai_sdk = ["assets/**/*"]` `package-data` glob in `pyproject.toml`.
- `native.py` — **live `NativeRuntime` adapter skeleton** (NOT a full backend; stage 1 + stage-2 LLM-service translation + stage-3 manifest translation). `NativeRuntime(Runtime)` (id `"native"`) is a factory for `NativeRuntimeSession(RuntimeSession)`, which wraps the wrapper `Agent` unchanged: translates `RuntimeOptions` → `Agent` kwargs (`_agent_kwargs_from_options`), drives `start()`/`stop()` lifecycle (`PENDING`→`ACTIVE`→`STOPPED`), routes `send()` onto `Agent.send()` (fire-and-forget queue), and exposes a non-blocking `events()` snapshot of lifecycle/notification/error events. Import-pure at module load (imports only `runtime` + the import-pure `errors`); the wrapper `Agent` and `lingtai.llm` service/builder are imported lazily on first `start()`. **Stage-2 LLM-service translation:** on the *default* factory path (`agent_factory` not injected), `start()` builds the `LLMService` the wrapper `Agent` requires via `_llm_service_from_options(options)` (lazy `lingtai.llm` import), passes it as `service`, and moves the secret-free LLM config from `session.deferred['llm']` into `session.applied['llm']` (`_public_llm_fields` strips `api_key`). Partial/absent `provider`/`model` raises `NativeRuntimeConfigurationError` *before* any agent is built (session stays `PENDING`) — the opaque missing-`service` `TypeError` no longer leaks (GLM stage-1 nit N2). An injected `agent_factory` **bypasses** service building (`_uses_default_factory` flag), so fakes/hosts boot without a real service. `api_key` is consumed into the service and never stored on the session, surfaced in events, or echoed in errors/reprs. **Stage-3 manifest translation (pure helpers):** `_llm_config_from_options` merges explicit `provider`/`model`/`base_url`/`api_key` over `manifest['llm']` (explicit wins, manifest fills gaps); `_max_rpm_from_options_or_manifest` resolves `max_rpm` (`extra['native']` → `extra` → `manifest` → `manifest['llm']`, default `0` = no SDK-imposed gating); `_context_window_from_options_or_manifest` resolves `context_window` (`manifest['llm']` → `manifest['context_limit']` → `extra['native']` → `extra`, else `None`). `_llm_service_from_options` builds from the merged config and, when a `manifest['llm']` block is present, plumbs recognized provider defaults (`api_compat`, `default_headers`, `compact_threshold`, `max_rpm`) through `build_provider_defaults_from_manifest_llm` **scoped to the merged provider**, plus `context_window`. `_sanitized_manifest` redacts `manifest['llm']['api_key']` from `session.deferred['manifest']` (the original `RuntimeOptions.manifest` is untouched). The merge runs before the provider/model presence check, so manifest-only config builds and partial config still raises a secret-free `NativeRuntimeConfigurationError`. **Stage-4 live event bridge:** when `bridge_events` is `True` (the default, threaded through `NativeRuntime` → `NativeRuntimeSession`), `start()` calls `_install_event_bridge(agent)` (`native.py:433`) right after the agent is built. The bridge wraps the agent's overridable hooks as *instance* attributes — purely additive, no kernel turn-loop change: `_on_tool_result_hook` → `TOOL_CALL` + `TOOL_RESULT` events (the original's return value passes through, so an intercepting host hook still short-circuits the turn); `_post_request` → a `TEXT` event for non-empty `result['text']`, a non-fatal `ERROR` per `result['errors']`, and a `USAGE` event sampled from `agent.get_token_usage()` (`_emit_turn_result` — `native.py:480`). The agent's life-state runs on its own loop thread, so it is **sampled** (not pushed): `events()` (`native.py:414`) calls `_sample_agent_state` (`native.py:499`), which reads `agent._state` and appends a `STATE` event only when it changed since the last read. Event appends and the last-sampled-state read/write are guarded by `self._lock` (the hooks fire on the agent's loop thread). Missing hooks/accessors are tolerated via `getattr` guards. `bridge_events=False` falls back to the stage-1 lifecycle/notification/error snapshot only.

## Connections

- **The kernel must never import this package.** Dependency flows one way: `lingtai_sdk` → (`lingtai`, `lingtai_kernel`). The SDK is a consumer of the other two, never a dependency of them.
- **Eager kernel, lazy wrapper.** `import lingtai_sdk` loads the dependency-light kernel only. Wrapper-backed names (`_LAZY_WRAPPER_EXPORTS`) import `lingtai` on first attribute access, keeping the wrapper's provider SDKs (anthropic/openai/google-genai/mcp/…) optional for pure-kernel consumers. Enforced by `tests/test_sdk_import_purity.py`.
- **Lazy SDK-internal names.** `NativeRuntime`/`NativeRuntimeSession` (`_LAZY_SDK_EXPORTS`) resolve from the import-pure `.native` module, NOT from the wrapper — accessing them and constructing a `NativeRuntime` stays provider-free; the wrapper `Agent` loads only when a session is started. Enforced by `tests/test_sdk_native_runtime.py::test_importing_native_and_constructing_runtime_is_pure`.
- `runtime.py`, `capabilities.py`, `capability_host.py`, `sdk_skill.py`, and `native.py` import **nothing from the `lingtai` wrapper** at module load (the purity tests forbid `lingtai`/`lingtai.`, not the dependency-light `lingtai_kernel`). `runtime.py` imports nothing from either; `capabilities.py` and `capability_host.py` import only the import-pure `.errors` (which re-exports `lingtai_kernel.types.UnknownToolError` — kernel, not wrapper); `capability_host.py` also imports `.capabilities`; `sdk_skill.py` imports only `importlib.resources` + `.capabilities` + `.capability_host` and reads its asset through `importlib.resources` (no wrapper, no filesystem-path guessing); `native.py` imports only the local `runtime` contract and `.errors`, deferring its wrapper `Agent` import to `start()`. Enforced by `tests/test_sdk_runtime_contract.py`, `tests/test_sdk_capabilities.py`, `tests/test_sdk_capability_host.py`, `tests/test_sdk_skill_bundle.py`, and `tests/test_sdk_native_runtime.py`.
- Compatibility is **by re-export, not re-implementation**: `_compat.DEPRECATIONS` records legacy→SDK path moves; the round-trip test asserts each pair resolves to the *same object* (`tests/test_sdk_compat.py`).

## Seeds vs. implementations

This package implements only the runtime *adapter* boundary; everything below it stops at the contract. **Present** (stage 1 + stage 2 + stage 3 + stage 4): the `NativeRuntime` skeleton in `native.py` — it wraps the existing `Agent`, drives its start/stop lifecycle, builds the `LLMService` the default factory needs from `RuntimeOptions` (stage 2), derives that config plus recognized provider defaults / `context_window` from `manifest['llm']` when explicit fields are absent (stage 3), and bridges the running agent's existing activity (tool calls/results, response text, usage, life-state) onto `RuntimeEvent`s via hook-wrapping + state sampling (stage 4). It still does not change the kernel turn loop or implement a non-native backend. The following remain **intentionally deferred** to later PRs and are NOT present here:

- A non-native backend (e.g. an Anthropic backend). (init.json-level manifest translation for the default-factory `LLMService` landed in stage 3 — `provider`/`model`/`base_url`/`api_key`, plus `api_compat`/`default_headers`/`compact_threshold`/`max_rpm` via `build_provider_defaults_from_manifest_llm`, and `context_window`. Migrating the privileged `system`/`psyche`/`soul` bundles into `BundleManifest` form is still out of scope.)
- A *streaming/blocking* event bridge — stage 4 added a **minimal** live bridge (tool calls/results, response text, usage, life-state via hook-wrapping + sampling on `events()` reads), but `events()` is still a non-blocking snapshot: there is no incremental token streaming, no async/await or generator that blocks until the next event, and no bridging of intra-turn provider chunks. Those richer surfaces remain deferred.
- Migration of the core `system` / `psyche` / `soul` bundles into `BundleManifest` form. The manifest/load/host *boundary* is proven end to end (`capabilities.load_manifest` + `capability_host.BundleHost`) against two harmless bundles: the synthetic `proof_bundle()` and — as of stage 6 — the *real, committed* `sdk_skill.sdk_skill_bundle()`, a non-privileged read-only skill asset hosted across tool/resource/prompt surfaces. This is genuine low-risk bundle adoption (a shipped asset, not synthetic echo), but it migrates **no privileged behavior**: `BundleHost` still refuses privileged/native-only bundles, and `system`/`psyche`/`soul` are NOT migrated. Wiring the wrapper's real *privileged* capabilities through manifests, and migrating the core bundles, remain later, higher-risk PRs.
- A distribution/package rename (the SDK ships inside the existing `lingtai` wheel for now).

See `docs/sdk/architecture-foundation.md` for the staged roadmap and rationale.

## State

`lingtai_sdk` is **stateless** — it writes nothing to disk and holds no runtime state. Its only side effects are the PEP 562 attribute cache populated in the module's own `globals()` when a lazy wrapper name is first accessed, and **read-only** reads of the committed `assets/` tree (`sdk_skill.load_sdk_skill()` reads `SKILL.md` via `importlib.resources`; it never writes).

## Notes

- Adding a public name: prefer re-exporting from `types.py`/`errors.py` (kernel-backed, eager) or `_LAZY_WRAPPER_EXPORTS` (wrapper-backed, lazy). Update `__all__` and, if it formalizes a legacy path, add a `_compat.Deprecation` entry.
- Do not import heavy provider SDKs at module load anywhere in this package; the import-purity tests will fail. Bare `google` (an ambient namespace stub pulled in transitively by `filelock`) is harmless and explicitly excluded from the purity check — only heavy submodules like `google.genai` count.
- Adding an SDK asset: drop it under `assets/<skill-name>/` (a skill-shaped layout), read it via `importlib.resources` (never a raw filesystem path), and rely on the `lingtai_sdk = ["assets/**/*"]` `package-data` glob so it ships in wheels. Keep assets read-only and import-pure. Update `assets/ANATOMY.md`.
