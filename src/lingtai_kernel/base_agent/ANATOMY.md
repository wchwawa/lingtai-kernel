# base_agent

> **Maintenance:** see the `lingtai-kernel-anatomy` skill. **Coding agents** update this file in the same commit as code changes. **LingTai agents** report drift as issues (mail or `discussions/<name>-patch.md`); do not silently fix.

Generic agent kernel. Single class `BaseAgent` with methods distributed across 6 helper modules. `__init__.py` retains the constructor, properties, state machine, and subclass-overridable hooks.

> **History note:** there used to be a 7th module, `soul_flow.py`. It was deleted in `1acd183` — soul-domain logic moved to `intrinsics/soul/flow.py`, and the wire-splice logic that lived in it became `tc_inbox.TCInbox.drain_into()`. Mention preserved here only because old patches/discussions reference it.

## Components

- `base_agent/__init__.py` — BaseAgent class definition (1531 lines after notification sync, pending-tool healing, and pass-through growth). Module-level helpers: `_format_stamina` (`base_agent/__init__.py:54`), `_build_identity_section` (`base_agent/__init__.py:65`). Class definition starts at `base_agent/__init__.py:150`; constructor at `base_agent/__init__.py:176`. Cluster pointers: `_wire_intrinsics` (`base_agent/__init__.py:457`), state machine (`_set_state` at `base_agent/__init__.py:561`, `_wake_nap` at `base_agent/__init__.py:590`, `_log` at `base_agent/__init__.py:595`), session persistence (`get_token_usage` at `base_agent/__init__.py:1421`, `get_chat_state`/`restore_chat`/`restore_token_state` at `base_agent/__init__.py:1433-1441`, `_save_chat_history` at `base_agent/__init__.py:1445`), subclass hooks (`_pre_request` at `base_agent/__init__.py:1510`, `_post_request` at `base_agent/__init__.py:1517`, `_on_tool_result_hook` at `base_agent/__init__.py:1523`). **Notification sync trio (the canonical `.notification/` consumer)**: `_strip_notification_prefix` (`base_agent/__init__.py:773`) — idempotent helper that removes a leading `notifications:\n…\n\n` block from a string; `_sync_notifications` (`base_agent/__init__.py:787`) — fingerprint-gated consumer dispatch over IDLE/ACTIVE/ASLEEP/STUCK/SUSPENDED, called from the heartbeat tick. **Wake mechanism**: on IDLE→IDLE strip+reinject and ASLEEP→IDLE wake, the synthesized `(ToolCallBlock, ToolResultBlock)` pair is appended via `_inject_notification_pair` (impersonating a voluntary `system(action="notification")` call from the agent's perspective), then `MSG_TC_WAKE` is posted to unblock the run loop. `_handle_tc_wake` drives one inference round off the existing wire — no fake user message, no meta prefix. From the LLM's viewpoint the wake is indistinguishable from the agent voluntarily calling `system(action="notification")`. `_inject_notification_pair` (`base_agent/__init__.py:982`) — splices the synthetic pair for IDLE / ASLEEP, returns False on `has_pending_tool_calls()`; `_inject_notification_meta` (`base_agent/__init__.py:1122`) — ACTIVE-state callback wired into `SessionManager.send()` via `notification_inject_fn`, prepends the JSON body onto the latest string-content `ToolResultBlock`. State attributes (`base_agent/__init__.py:373-391`): `_notification_fp`, `_notification_block_id`, `_pending_notification_meta`, `_pending_notification_fp`. Empty-state sync (`base_agent/__init__.py:837-844`) clears pending ACTIVE metadata so a dismissed channel cannot be re-injected later. See root `ANATOMY.md` "Notifications" for the full architecture. **Legacy tc_inbox plumbing (dormant, slated for Phase 3 removal)**: `_drain_tc_inbox` (`base_agent/__init__.py:650`), `_install_drain_hook` (`base_agent/__init__.py:681`), `_drain_tc_inbox_for_hook` (`base_agent/__init__.py:740`) — still installed and called, but the queue is always empty in production now that all producers write `.notification/` files instead. Plus 46 pass-through stubs to submodules (~92 lines of boilerplate).
- `base_agent/identity.py` — Identity, manifest, and status (150 lines). `_set_name` (`base_agent/identity.py:11`), `_set_nickname` (`base_agent/identity.py:24`), `_update_identity` (`base_agent/identity.py:30`), `_build_manifest` (`base_agent/identity.py:50`), `_status` (`base_agent/identity.py:83`).
- `base_agent/lifecycle.py` — Heartbeat, signal-file detection, start/stop, refresh (510 lines after the heartbeat-lifetime fix). `_start` (`base_agent/lifecycle.py:15`), `_stop` (`base_agent/lifecycle.py:84`), `_start_heartbeat` (`base_agent/lifecycle.py:124`), `_stop_heartbeat` (`base_agent/lifecycle.py:138`), `_heartbeat_loop` (`base_agent/lifecycle.py:152`), `_perform_refresh` (`base_agent/lifecycle.py:371`), `_can_fallback_preset` (`base_agent/lifecycle.py:461`), `_check_rules_file` (`base_agent/lifecycle.py:475`). **Heartbeat-file lifetime contract**: `.agent.heartbeat` is created on `_start_heartbeat` and unlinked at the **end** of `_stop` (just before `release_lock`), so external observers — TUI launcher, `lingtai-tui list`, `lingtai-tui purge` — see the workdir as live for the entire teardown window. The heartbeat loop intentionally keeps writing fresh timestamps after `agent._shutdown.is_set()` (loop predicate is `agent._heartbeat_thread is not None`); only signal-file detection is gated on `_shutdown`. Without this contract, a quick suspend→relaunch would race a still-living interpreter into the same workdir (the kernel's flock prevents data corruption but two Python processes still appear in `ps`). **Notification sync poll**: the heartbeat loop calls `agent._sync_notifications()` (`base_agent/lifecycle.py:334`), wrapped in a `try/except` that logs warnings without aborting the tick. This is the primary trigger for `.notification/` filesystem reads. **Refresh dispatch contract**: `_perform_refresh` calls `agent._build_launch_cmd()` as a bound method — must NOT be re-introduced as a module-level function (would shadow the `Agent` subclass override and silently no-op every user refresh). See issue #7 — confirmed in vivo against deepseek_pro 2026-05-05 (`refresh_start → refresh_no_launch_cmd` in 0.78ms, PID stable across 5 "refreshes").
- `base_agent/turn.py` — Turn engine: main loop, message dispatch, AED, response processing, LLM-hang watchdog (1026 lines). LLM-hang machinery: `_on_llm_hang` (`base_agent/turn.py:55`), `_send_with_watchdog` (`base_agent/turn.py:152`). Run loop: `_run_loop` (`base_agent/turn.py:206`), `_concat_queued_messages` (`base_agent/turn.py:417`), `_handle_message` (`base_agent/turn.py:463`), `_handle_request` (`base_agent/turn.py:571`), `_handle_tc_wake` (`base_agent/turn.py:625`) — **post-redesign: drives one inference round off the existing canonical interface via `session.send(None)` when the tail is `user[ToolResultBlock]` (the shape produced by `_inject_notification_pair`).** Legacy `tc_inbox.drain()` items are still spliced in the old way (call appended here, result passed through send) at the top of the function for back-compat, but the queue is always empty in production. The wire-drive path is what makes notifications visible to the LLM: no fake user message, no meta prefix. **AED state transitions** (within `_run_loop`): on each retry attempt the loop first closes pending tool calls with synthesized results (`turn.py:325-333`), then sets STUCK (`turn.py:335`) before rebuilding the session and setting ACTIVE again right before re-entering `_handle_message` with the recovery message (`turn.py:369`). Without that ACTIVE flip, a recovered turn that does long work (file writes, mail, bash) leaves `.agent.json:state` reading "stuck" until the post-turn sweep at `turn.py:388-389` finally writes `sleep_state` (IDLE) — visible to the TUI/portal as a stale "滞" badge while the agent is in fact busy. `_get_guard_limits` (`base_agent/turn.py:781`), `_process_response` (`base_agent/turn.py:871`).
- `base_agent/tools.py` — Tool surface: schemas, dispatch, registry (161 lines). `_dispatch_tool` (`base_agent/tools.py:13-33`), `_refresh_tool_inventory_section` (`base_agent/tools.py:35-50`), `_build_tool_schemas` (`base_agent/tools.py:52-100`), `_add_tool` (`base_agent/tools.py:102-131`), `_remove_tool` (`base_agent/tools.py:133-142`), `_override_intrinsic` (`base_agent/tools.py:144-157`), `_has_capability` (`base_agent/tools.py:159-161`).
- `base_agent/prompt.py` — System prompt building, flushing, updating (55 lines). `_build_system_prompt` (`base_agent/prompt.py:9-16`), `_build_system_prompt_batches` (`base_agent/prompt.py:18-32`), `_flush_system_prompt` (`base_agent/prompt.py:34-42`), `_update_system_prompt` (`base_agent/prompt.py:44-55`).
- `base_agent/messaging.py` — Mail arrival, notification producers, outbound messaging (225 lines after notification migration and lock-safe system event merging). `_on_mail_received` (`base_agent/messaging.py:11-18`), `_on_normal_mail` (`base_agent/messaging.py:21-49`) — calls `_wake_nap` for sub-second sync latency and republishes the unread digest; **does NOT post `MSG_TC_WAKE` itself** — wake transitions are owned by the kernel notification sync mechanism. **`_rerender_unread_digest`** (`base_agent/messaging.py:52-106`) — email producer; calls `system.publish_notification(workdir, "email", header=…, icon="📧", data={count, newest_received_at, digest})` when count > 0, else `clear_notification(workdir, "email")`. **`_enqueue_system_notification`** (`base_agent/messaging.py:109-194`) — system events producer; merges `(event_id, source, ref_id, body, at)` into the events list inside `.notification/system.json`, capped at 20 newest. **Lazy `threading.Lock`** (`agent._system_notification_lock`) guards the read-modify-write merge so concurrent bursts (e.g. mail bounces) don't lose events. Only `system.json` needs the lock; `email.json` and `soul.json` recompute full state on every publish (no merge). Returns an `event_id` for back-compat with callers that expected a notif_id; per-id lifecycle is gone under the new model. `_notify` (`base_agent/messaging.py:197-204`), `_mail` (`base_agent/messaging.py:207-213`), `_send` (`base_agent/messaging.py:215-225`).

## Connections

- All submodules are leaves — they import nothing from `base_agent/`. Cross-module communication is mediated through the `agent` parameter (the BaseAgent instance).
- `__init__.py` imports from all submodules lazily (inside method bodies) to avoid circular imports at load time. Soul-flow pass-throughs now import from `intrinsics/soul/flow.py` and `intrinsics/soul/inquiry.py`.
- `__init__.py` → `notifications.py` (via `_sync_notifications` consuming `notification_fingerprint`/`collect_notifications`).
- `__init__.py` constructs `SessionManager` with `notification_inject_fn=self._inject_notification_meta` (`base_agent/__init__.py:464`), so ACTIVE-state notification meta gets prepended to the latest `ToolResultBlock` at request-send time inside `SessionManager.send()`.
- `lifecycle.py` → `intrinsics/soul/flow.py` (via `agent._start_soul_timer()`, `agent._cancel_soul_timer()`), `intrinsics/soul/inquiry.py` (via `agent._run_inquiry()`), and `agent._sync_notifications()` on every heartbeat tick.
- `turn.py` → `intrinsics/soul/flow.py` (via `agent._cancel_soul_timer()`), `intrinsics/soul/inquiry.py` (via `agent._run_inquiry()`). The `tc_inbox` drain calls in `_handle_request`, the pre-request hook, and `_handle_tc_wake` are still wired but always find an empty queue under the `.notification/` redesign.
- `messaging.py` → `intrinsics/system` (via `publish_notification`/`clear_notification` for both `_rerender_unread_digest` and `_enqueue_system_notification`), and `notifications.collect_notifications` for the system-events RMW merge.
- `prompt.py` → `tools.py` (via `agent._refresh_tool_inventory_section()`).
- All submodules read agent attributes (`agent._config`, `agent._session`, `agent._chat`, `agent._notification_fp`, `agent._notification_block_id`, `agent._pending_notification_meta`, etc.) — the agent object is the shared state.
- `messaging.py` imports `from ..message import _make_message, MSG_REQUEST, MSG_TC_WAKE` and `from ..i18n import t as _t`. (`MSG_TC_WAKE` is no longer posted from `messaging.py` itself; the kernel sync owns wake transitions.)
- `tools.py` imports `from ..intrinsics import ALL_INTRINSICS`, `from ..llm import FunctionSchema`, `from ..i18n import t as _t`.
- `turn.py` imports `from ..state import AgentState`, `from ..loop_guard import LoopGuard`, `from ..tool_executor import ToolExecutor`.
- `lifecycle.py` imports `from ..state import AgentState`, `from ..token_ledger import sum_token_ledger`.

## State

- `identity.py` mutates `.agent.json` (manifest) and `system/system.md` (identity prompt section) via `_build_manifest` (`base_agent/identity.py:50-80`) and `_update_identity` (`base_agent/identity.py:30-47`).
- `lifecycle.py` writes `.agent.heartbeat` (`base_agent/lifecycle.py:173-177`; the file's lifetime equals the Python process's lifetime — see "Heartbeat-file lifetime contract" above), consumes signal files (`.interrupt`, `.refresh`, `.suspend`, `.sleep`, `.prompt`, `.clear`, `.inquiry`, `.rules`), and reads `.notification/*.json` via `_sync_notifications` on every tick.
- `turn.py` writes `history/chat_history.jsonl` via `_save_chat_history` (delegated to `__init__.py`), writes `.status.json` and `logs/token_ledger.jsonl`.
- `messaging.py` writes `.notification/email.json` (via `_rerender_unread_digest`) and merges into `.notification/system.json` (via `_enqueue_system_notification`, RMW under a per-agent `threading.Lock`).
- `__init__.py` writes the notification `(call, result)` pair into `chat_history.jsonl` via `_save_chat_history(ledger_source="notification_sync")` after a successful `_inject_notification_pair`. The same call also emits a `notification_pair_injected` event to `events.jsonl` carrying `call_id`, `sources`, `summary`, **and `meta`** (the same `build_meta(agent)` dict — `current_time`, `context.{system_tokens,history_tokens,usage}`, `stamina_left_seconds`, `injection_seq` — that the agent itself sees in the synthesized tool-result body). The TUI's `parseEvent` reads the `meta` block and renders it as a faint footer line under the notification banner; older events.jsonl rows that pre-date this addition simply omit `meta` and the renderer drops the footer. See `lingtai/tui/internal/fs/session.go::parseEvent` notification branch and issue #40.

## Notes

- **`__init__.py` is 1531 lines.** This is intentional — the constructor (230 lines), properties, state machine, hooks, cross-cutting infrastructure (`_save_chat_history`, `_log`, the notification sync trio `_sync_notifications`/`_inject_notification_pair`/`_inject_notification_meta`, the legacy tc_inbox drain trio), and 46 pass-through stubs all live here. The package is not "thin shell + 6 leaves" — it's "1500-line core + 6 specialized helpers." Future readers should not try to extract more from `__init__.py` thinking it should be smaller; the remaining code is genuinely cross-cutting or bound to the class definition.
- **Pass-through pattern.** Each extracted method becomes a 2-line stub in `__init__.py` (lazy import + call). This preserves the `BaseAgent` class interface while the implementation lives in submodules. The `self` → `agent` conversion is mechanical but `_heartbeat_loop` (183 lines, 15+ cross-cluster calls) deserves extra-careful review.
- **Subclass overrides stay on `__init__.py`.** `_activate_preset`, `_activate_default_preset`, `_build_launch_cmd`, `_pre_request`, `_post_request`, `_on_tool_result_hook`, `_cpr_agent` are all overridden by the `Agent` subclass in the wrapper package. They must remain as methods on `BaseAgent`.
