# base_agent

> **Maintenance:** see the `lingtai-kernel-anatomy` skill. **Coding agents** update this file in the same commit as code changes. **LingTai agents** report drift as issues/mail/PR proposals; do not silently fix.

Generic agent kernel. Single class `BaseAgent` with methods distributed across 6 helper modules. `__init__.py` retains the constructor, properties, state machine, and subclass-overridable hooks.

> **History note:** there used to be a 7th module, `soul_flow.py`. It was deleted in `1acd183` ÔÇö soul-domain logic moved to `intrinsics/soul/flow.py`, and the wire-splice logic that lived in it became `tc_inbox.TCInbox.drain_into()`. Mention preserved here only because old patches/discussions reference it.

- `base_agent/lifecycle.py` ÔÇö Heartbeat, signal-file detection, start/stop, refresh, nudge dispatch (787 lines). `_start` (`base_agent/lifecycle.py:29`), `_stop` (`base_agent/lifecycle.py:98`), `_shutdown_daemon_runtime` (`base_agent/lifecycle.py:144`), `_start_heartbeat` (`base_agent/lifecycle.py:176`), `_stop_heartbeat` (`base_agent/lifecycle.py:190`), `_heartbeat_loop` (`base_agent/lifecycle.py:204`), `_perform_refresh` (`base_agent/lifecycle.py:469`), `_can_fallback_preset` (`base_agent/lifecycle.py:738`), `_check_rules_file` (`base_agent/lifecycle.py:752`). **Daemon teardown contract**: `_stop` calls `_shutdown_daemon_runtime` after joining the main loop but before closing session services, stopping heartbeat, or releasing the workdir lock. If the daemon capability is present, `DaemonManager.shutdown_for_agent_stop()` kills tracked CLI process groups and shuts down executor pools while `.agent.heartbeat` is still fresh, preventing refresh watchers from seeing a dead heartbeat/clear lock while the old Python interpreter is still held alive by daemon workers. **Heartbeat-file lifetime contract**: `.agent.heartbeat` is created on `_start_heartbeat` and unlinked at the **end** of `_stop` (just before `release_lock`), so external observers ÔÇö TUI launcher, `lingtai-tui list`, `lingtai-tui purge` ÔÇö see the workdir as live for the entire teardown window. The heartbeat loop intentionally keeps writing fresh timestamps after `agent._shutdown.is_set()` (loop predicate is `agent._heartbeat_thread is not None`); only signal-file detection is gated on `_shutdown`. Without this contract, a quick suspendÔåÆrelaunch would race a still-living interpreter into the same workdir (the kernel's flock prevents data corruption but two Python processes still appear in `ps`). **Nudge dispatch**: just before the notification sync poll, the heartbeat loop calls `nudge.run_checks(agent)` (`base_agent/lifecycle.py:~320`), wrapped in `try/except` that logs warnings. Each registered nudge check (today: `kernel_version`) throttles itself internally and writes to `.notification/nudge.json` via `nudge.upsert` / `nudge.remove`. See `nudge/ANATOMY.md`. **Notification sync poll**: the heartbeat loop then calls `agent._sync_notifications()`, wrapped in a `try/except` that logs warnings without aborting the tick. This is the primary trigger for `.notification/` filesystem reads (including the `nudge.json` file the dispatch may have just written). **Refresh dispatch contract**: `_perform_refresh` calls `agent._build_launch_cmd()` as a bound method ÔÇö must NOT be re-introduced as a module-level function (would shadow the `Agent` subclass override and silently no-op every user refresh). See issue #7 ÔÇö confirmed in vivo against deepseek_pro 2026-05-05 (`refresh_start ÔåÆ refresh_no_launch_cmd` in 0.78ms, PID stable across 5 "refreshes"). **Refresh handshake self-sufficiency**: `_perform_refresh` is called from three sites ÔÇö the heartbeat (`.refresh` signal-file path), the `system(action='refresh')` intrinsic, and the AED preset-fallback in `turn.py`. All three reach the same code without coordination, so `_perform_refresh` normalizes the on-disk handshake (`.refresh.taken` ack file present and `.refresh` cleared) before spawning the watcher subprocess, and sets `agent._cancel_event` + `agent._shutdown` after spawn so the lock-release phase completes regardless of caller. The `refresh_deferred_relaunch` event records `handshake` Ôêê {`preexisting_taken`, `renamed_refresh`, `synthesized_direct_call`, `synthesized_after_rename_failed`, `ack_write_failed`} for telemetry; if the ack cannot be established, `_perform_refresh` logs `refresh_ack_failed` and returns before spawning a watcher or setting shutdown. Direct callers thus never trip the watcher's `phase='ack'` or `phase='lock'` timeouts in the normal established-ack path. **Refresh stale-process recovery**: if the watcher relaunch is rejected by the CLI duplicate-process guard (`another lingtai agent is already running`) and the blocking process is still a same-workdir `lingtai run` with no fresh heartbeat, the watcher logs `refresh_watcher_stale_duplicate_*`, sends `SIGTERM`, escalates to `SIGKILL` after a short grace period, and retries. A fresh heartbeat blocks cleanup and logs `refresh_watcher_duplicate_alive` instead.
- `base_agent/turn.py` ÔÇö Turn engine: main loop, message dispatch, AED, response processing, and ACTIVE-turn tool-call progress-meter wiring (1606 lines). Transient provider retry helpers (`base_agent/turn.py:47-114`) classify SDK/network 5xx-ish failures and `EmptyLLMResponseError` before they spend AED budget. `WorkerStillRunningError` (from `llm_utils.py`) is handled directly in the AED loop: the worker interface is marked poisoned (see `worker_recovery.py`), a bounded/redacted recovery artifact is written, a high-priority `kernel.llm_worker_hang` notification is published, a skip-save forced refresh is requested, and the agent is put STUCKÔåÆASLEEP with `skip_post_turn_save=True` so the in-process `ChatInterface` is never mutated while the worker future is still alive ÔÇö refresh/relaunch brings up a fresh interface from the last safe on-disk history. A residual poison guard at the top of the AED `try` also short-circuits any later turn that reaches the loop while poisoned. No filesystem sentinel ÔÇö the previous `.llm_hang` watchdog/sentinel system was removed because its 120s timer false-fired on slow thinking-model turns and the sentinel could block wake-from-asleep even after a successful refresh. Run loop: `_run_loop` (`base_agent/turn.py:461`), `_concat_queued_messages` (`base_agent/turn.py:727`), `_handle_message` (`base_agent/turn.py:773`), `_handle_request` (`base_agent/turn.py:887`), `_handle_tc_wake` (`base_agent/turn.py:935`) ÔÇö **post-redesign: drives one inference round off the existing canonical interface via `session.send(None)` when the tail is `user[ToolResultBlock]` (the shape produced by `_inject_notification_pair`).** Legacy `tc_inbox.drain()` items are still spliced in the old way (call appended here, result passed through send) at the top of the function for back-compat, but the queue is always empty in production. The wire-drive path is what makes notifications visible to the LLM: no fake user message, no meta prefix. **AED state transitions** (within `_run_loop`): transient provider failures first get up to three local retry attempts with 1/2/4s backoff, pending-tool-call healing, and `system.stuck_revive` retry messages, but without incrementing `aed_attempts` or rebuilding the session (`base_agent/turn.py:561-589`). Counted AED attempts then close pending tool calls with synthesized results (`base_agent/turn.py:591-601`), set STUCK (`base_agent/turn.py:603`) before rebuilding the session (`base_agent/turn.py:629-631`), and set ACTIVE again right before re-entering `_handle_message` with the recovery message (`base_agent/turn.py:634-635`). Without that ACTIVE flip, a recovered turn that does long work (file writes, mail, bash) leaves `.agent.json:state` reading "stuck" until the post-turn sweep at `base_agent/turn.py:636-658` finally writes `sleep_state` (IDLE) ÔÇö visible to the TUI/portal as a stale "µ╗×" badge while the agent is in fact busy. `_get_guard_limits` (`base_agent/turn.py:1103`), `_process_response` (`base_agent/turn.py:1241`; empty response raises at `base_agent/turn.py:1289`). **Tool errors stay in the normal result flow**: `_process_response` no longer has a repeated-identical-tool-error hard-stop or `session.send(str)` continuation path. Tool execution errors remain ordinary `ToolResultBlock` payloads that are sent back through `session.send(tool_results)`; `ToolExecutor` enriches those error payloads with model-visible `tool_error` recovery metadata (`tool_executor.py:146-235`) while the generic tool-loop / emergency limits remain the only backstop for true infinite loops.

- All submodules are leaves ÔÇö they import nothing from `base_agent/`. Cross-module communication is mediated through the `agent` parameter (the BaseAgent instance).

## State

- `identity.py` mutates `.agent.json` (manifest) and `system/system.md` (identity prompt section) via `_build_manifest` (`base_agent/identity.py:58-97`) and `_update_identity` (`base_agent/identity.py:30-47`). The manifest now includes a sanitized `llm` block (`provider`, `model`, `base_url`) read from the live `LLMService` via `_safe_llm_from_service` (`base_agent/identity.py:100-177`). The wrapper subclass extends with `preset` (active/default/allowed) read from `init.json` and re-applies the LLM safelist for defense-in-depth ÔÇö see `src/lingtai/agent.py`. The identity prose renderer (`base_agent/__init__.py:139-181` inside `_build_identity_section`) folds both blocks into the cached system-prompt prefix only when present, so bare BaseAgent instances stay quiet.
- `lifecycle.py` writes `.agent.heartbeat` (`base_agent/lifecycle.py:207-211`; the file's lifetime equals the Python process's lifetime ÔÇö see "Heartbeat-file lifetime contract" above), consumes signal files (`.interrupt`, `.refresh`, `.suspend`, `.sleep`, `.prompt`, `.clear`, `.inquiry`, `.rules`), and reads `.notification/*.json` via `_sync_notifications` on every tick.
- `turn.py` writes `history/chat_history.jsonl` via `_save_chat_history` (delegated to `__init__.py`), writes `.status.json` and `logs/token_ledger.jsonl`, and mutates the live notification holder on dict-shaped tool results by adding/removing the canonical `_meta.notifications` + `_meta.notification_guidance` payload. It also constructs the per-turn `ToolExecutor` with `working_dir=agent._working_dir` (`base_agent/turn.py:690-704`, `774-785`) so the executor can spill oversized tool results into `<workdir>/tmp/tool-results/` ÔÇö see `tool_result_artifacts.py:spill_oversized_result`. Notification-pair injection in `__init__.py` bypasses `ToolExecutor` and therefore the preventive cap, by design. **Retroactive compaction (issue #144)**: before either AED retry branch, `_compact_history_before_retry` walks `_session.chat.interface._entries` and rewrites already-committed `ToolResultBlock.content` larger than the 5K retroactive cap into the same `lingtai_tool_result_spill` manifest shape ÔÇö entries / ids / pairing / `synthesized` flag are preserved. The helper logs a single bounded summary event `aed_history_compacted` (`scanned_blocks`, `compacted_blocks`, `original_chars_total`, `replacement_chars_total`, up to 16 `artifact_paths`) and, when at least one block was rewritten, calls `_save_chat_history(ledger_source="retroactive_compaction")` so the persisted history matches the wire before the rebuilt session replays it. **Over-window routing**: errors whose `str(exc)` matches `_OVER_WINDOW_MSG_FRAGMENTS` (context-window / prompt-too-long / token-count phrasing across providers) short-circuit the transient retry loop and take the deterministic branch with `source="aed_over_window"`, since retrying on the same wire would refire the same error. **WorkerStillRunningError is exempt** ÔÇö that branch puts the agent ASLEEP without touching ChatInterface, so compaction never runs (it would race the still-alive worker future). The helper itself is exception-safe; any failure logs `tool_result_compaction_failed` and never re-raises. A hard pre-send gate (run compaction whenever the wire exceeds a threshold *before* attempting send) is left as a TODO in `_run_loop`.
- `messaging.py` writes `.notification/email.json` (via `_rerender_unread_digest`) and merges into `.notification/system.json` (via `_enqueue_system_notification`, RMW under a per-agent `threading.Lock`).
- `worker_recovery.py` writes `history/unfinished_turns/worker_still_running_*.json` (bounded/redacted unfinished-turn artifacts; atomic tmp+rename) and marks `prompt_injected_at` on the first safe request after restart. The poison flag and related `_llm_worker_*` fields are process-local in-memory state only (never persisted). `lifecycle._start` reads open artifacts via `rehydrate_worker_hang_recovery` to re-publish the high-priority notification after relaunch.
- `_inject_notification_pair()` still emits compact `notification_pair_injected` metadata for session summaries, while `_log_notification_block_injected()` records the durable actual-block history (`mode`, `call_id`, `sources`, full `_meta` envelope with `tool_meta`, `agent_meta`, `guidance`, `notifications`, `notification_guidance`) for TUI `/notification` and log readers (`__init__.py:1441`, `__init__.py:1476`).

## Notes

- **`__init__.py` is large.** This is intentional ÔÇö the constructor, properties, state machine, hooks, cross-cutting infrastructure (`_save_chat_history`, `_log`, notification sync, legacy tc_inbox drain), and pass-through stubs all live here. The package is not "thin shell + 6 leaves"; the remaining code is genuinely cross-cutting or bound to the class definition.
- **Pass-through pattern.** Each extracted method becomes a 2-line stub in `__init__.py` (lazy import + call). This preserves the `BaseAgent` class interface while the implementation lives in submodules. The `self` ÔåÆ `agent` conversion is mechanical but `_heartbeat_loop` (183 lines, 15+ cross-cluster calls) deserves extra-careful review.
- **Subclass overrides stay on `__init__.py`.** `_activate_preset`, `_activate_default_preset`, `_build_launch_cmd`, `_pre_request`, `_post_request`, `_on_tool_result_hook`, `_cpr_agent` are all overridden by the `Agent` subclass in the wrapper package. They must remain as methods on `BaseAgent`.
