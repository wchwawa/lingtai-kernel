# base_agent

> **Maintenance:** see the `lingtai-kernel-anatomy` skill. **Coding agents** update this file in the same commit as code changes. **LingTai agents** report drift as issues (mail or `discussions/<name>-patch.md`); do not silently fix.

Generic agent kernel. Single class `BaseAgent` with methods distributed across 6 helper modules. `__init__.py` retains the constructor, properties, state machine, and subclass-overridable hooks.

> **History note:** there used to be a 7th module, `soul_flow.py`. It was deleted in `1acd183` — soul-domain logic moved to `intrinsics/soul/flow.py`, and the wire-splice logic that lived in it became `tc_inbox.TCInbox.drain_into()`. Mention preserved here only because old patches/discussions reference it.

## Components

- `base_agent/__init__.py` — BaseAgent class definition (~1480 lines after issue #82/#83 notification cleanup). Module-level helpers: `_format_stamina` (`base_agent/__init__.py:48`), `_build_identity_section` (`base_agent/__init__.py:59`), `_identity_scalar` (`base_agent/__init__.py:178`). The identity-section renderer reads `manifest_data["llm"]` (provider/model/base_url) and `manifest_data["preset"]` (active/default) when present and renders a short "running on" + "active preset" prose line so the agent sees its current runtime — see issue #78. Class definition starts at `base_agent/__init__.py:192`; constructor at `base_agent/__init__.py:218`. Cluster pointers: `_wire_intrinsics` (`base_agent/__init__.py:495`), state machine (`_set_state` at `base_agent/__init__.py:599`), `_wake_nap` (`base_agent/__init__.py:628`), `_note_notification_deferred_active` + `_log` (`base_agent/__init__.py:722-807`), session persistence (`get_token_usage` / `get_chat_state` / `restore_chat` / `restore_token_state` / `_save_chat_history` near `base_agent/__init__.py:1371-1409`), and subclass hooks near the file tail. **Notification sync/live holder**: `_sync_notifications` (`base_agent/__init__.py:817`) — fingerprint-gated consumer dispatch over IDLE/ACTIVE/ASLEEP/STUCK/SUSPENDED, called from the heartbeat tick; `_inject_notification_pair` (`base_agent/__init__.py:1019-1184`) — splices the synthetic `system(action="notification")` pair for IDLE / ASLEEP and returns False on `has_pending_tool_calls()`. Wake mechanism: IDLE/ASLEEP delivery appends the synthesized `(ToolCallBlock, ToolResultBlock)` pair, posts `MSG_TC_WAKE`, and `_handle_tc_wake` drives one inference round off the existing wire. ACTIVE delivery moves the same canonical `notifications` + `_notification_guidance` payload onto the latest dict-shaped tool result via `meta_block.attach_active_notifications` in `turn.py` (`base_agent/turn.py:926-937`), stripping/skeletonizing the previous holder; while a turn is still ACTIVE, repeated heartbeat deferrals are counted for watchdog status but `notification_deferred_active` logging is deduplicated by filesystem fingerprint in `_note_notification_deferred_active`. The injected wire body carries a top-level `_notification_guidance` plus a source-specific `_notification_guidance` field under each entry in `notifications` (kernel safety framing, separate from any producer-supplied `instructions` field). State attributes: `_notification_fp`, `_notification_block_id`, `_notification_inject_seq`, `_notification_live_holder` (initialized at `base_agent/__init__.py:415-439`). See root `ANATOMY.md` "Notifications" for the full architecture. **Legacy tc_inbox plumbing (dormant, slated for Phase 3 removal)**: `_drain_tc_inbox` (`base_agent/__init__.py:688`), `_install_drain_hook` (`base_agent/__init__.py:719`), `_drain_tc_inbox_for_hook` (`base_agent/__init__.py:778`) — still installed and called, but the queue is always empty in production now that all producers write `.notification/` files instead. Plus pass-through stubs to submodules.

- `base_agent/identity.py` — Identity, manifest, and status (~247 lines after issue #78). `_set_name` (`base_agent/identity.py:11`), `_set_nickname` (`base_agent/identity.py:24`), `_update_identity` (`base_agent/identity.py:30`), `_LLM_PUBLIC_KEYS` safelist constant (`base_agent/identity.py:55`), `_build_manifest` (`base_agent/identity.py:58`), `_safe_llm_from_service` (`base_agent/identity.py:100`) — surfaces a sanitized `llm` block (provider/model/base_url, safelisted) from the live `LLMService` instance for `.agent.json` and the identity prompt section (issue #78). Never reads from `init.json` (kernel stays preset-oblivious); preset metadata is added by the wrapper's `Agent._build_manifest` override. `_status` (`base_agent/identity.py:180`).
- `base_agent/lifecycle.py` — Heartbeat, signal-file detection, start/stop, refresh, nudge dispatch (787 lines). `_start` (`base_agent/lifecycle.py:29`), `_stop` (`base_agent/lifecycle.py:98`), `_shutdown_daemon_runtime` (`base_agent/lifecycle.py:144`), `_start_heartbeat` (`base_agent/lifecycle.py:176`), `_stop_heartbeat` (`base_agent/lifecycle.py:190`), `_heartbeat_loop` (`base_agent/lifecycle.py:204`), `_perform_refresh` (`base_agent/lifecycle.py:469`), `_can_fallback_preset` (`base_agent/lifecycle.py:738`), `_check_rules_file` (`base_agent/lifecycle.py:752`). **Daemon teardown contract**: `_stop` calls `_shutdown_daemon_runtime` after joining the main loop but before closing session services, stopping heartbeat, or releasing the workdir lock. If the daemon capability is present, `DaemonManager.shutdown_for_agent_stop()` kills tracked CLI process groups and shuts down executor pools while `.agent.heartbeat` is still fresh, preventing refresh watchers from seeing a dead heartbeat/clear lock while the old Python interpreter is still held alive by daemon workers. **Heartbeat-file lifetime contract**: `.agent.heartbeat` is created on `_start_heartbeat` and unlinked at the **end** of `_stop` (just before `release_lock`), so external observers — TUI launcher, `lingtai-tui list`, `lingtai-tui purge` — see the workdir as live for the entire teardown window. The heartbeat loop intentionally keeps writing fresh timestamps after `agent._shutdown.is_set()` (loop predicate is `agent._heartbeat_thread is not None`); only signal-file detection is gated on `_shutdown`. Without this contract, a quick suspend→relaunch would race a still-living interpreter into the same workdir (the kernel's flock prevents data corruption but two Python processes still appear in `ps`). **Nudge dispatch**: just before the notification sync poll, the heartbeat loop calls `nudge.run_checks(agent)` (`base_agent/lifecycle.py:~320`), wrapped in `try/except` that logs warnings. Each registered nudge check (today: `kernel_version`) throttles itself internally and writes to `.notification/nudge.json` via `nudge.upsert` / `nudge.remove`. See `nudge/ANATOMY.md`. **Notification sync poll**: the heartbeat loop then calls `agent._sync_notifications()`, wrapped in a `try/except` that logs warnings without aborting the tick. This is the primary trigger for `.notification/` filesystem reads (including the `nudge.json` file the dispatch may have just written). **Refresh dispatch contract**: `_perform_refresh` calls `agent._build_launch_cmd()` as a bound method — must NOT be re-introduced as a module-level function (would shadow the `Agent` subclass override and silently no-op every user refresh). See issue #7 — confirmed in vivo against deepseek_pro 2026-05-05 (`refresh_start → refresh_no_launch_cmd` in 0.78ms, PID stable across 5 "refreshes"). **Refresh handshake self-sufficiency**: `_perform_refresh` is called from three sites — the heartbeat (`.refresh` signal-file path), the `system(action='refresh')` intrinsic, and the AED preset-fallback in `turn.py`. All three reach the same code without coordination, so `_perform_refresh` normalizes the on-disk handshake (`.refresh.taken` ack file present and `.refresh` cleared) before spawning the watcher subprocess, and sets `agent._cancel_event` + `agent._shutdown` after spawn so the lock-release phase completes regardless of caller. The `refresh_deferred_relaunch` event records `handshake` ∈ {`preexisting_taken`, `renamed_refresh`, `synthesized_direct_call`, `synthesized_after_rename_failed`, `ack_write_failed`} for telemetry; if the ack cannot be established, `_perform_refresh` logs `refresh_ack_failed` and returns before spawning a watcher or setting shutdown. Direct callers thus never trip the watcher's `phase='ack'` or `phase='lock'` timeouts in the normal established-ack path. **Refresh stale-process recovery**: if the watcher relaunch is rejected by the CLI duplicate-process guard (`another lingtai agent is already running`) and the blocking process is still a same-workdir `lingtai run` with no fresh heartbeat, the watcher logs `refresh_watcher_stale_duplicate_*`, sends `SIGTERM`, escalates to `SIGKILL` after a short grace period, and retries. A fresh heartbeat blocks cleanup and logs `refresh_watcher_duplicate_alive` instead.
- `base_agent/turn.py` — Turn engine: main loop, message dispatch, AED, response processing (941 lines). Transient provider retry helpers (`base_agent/turn.py:42-115`) classify SDK/network 5xx-ish failures and `EmptyLLMResponseError` before they spend AED budget. `WorkerStillRunningError` (from `llm_utils.py`) is handled directly in the AED loop (`base_agent/turn.py:209-223`): the agent is put to STUCK→ASLEEP with `skip_post_turn_save=True` so the in-process `ChatInterface` is not mutated while the worker future is still alive — a refresh brings up a fresh interface. No filesystem sentinel — the previous `.llm_hang` watchdog/sentinel system was removed because its 120s timer false-fired on slow thinking-model turns and the sentinel could block wake-from-asleep even after a successful refresh. Run loop: `_run_loop` (`base_agent/turn.py:130`), `_concat_queued_messages` (`base_agent/turn.py:350`), `_handle_message` (`base_agent/turn.py:396`), `_handle_request` (`base_agent/turn.py:487`), `_handle_tc_wake` (`base_agent/turn.py:534`) — **post-redesign: drives one inference round off the existing canonical interface via `session.send(None)` when the tail is `user[ToolResultBlock]` (the shape produced by `_inject_notification_pair`).** Legacy `tc_inbox.drain()` items are still spliced in the old way (call appended here, result passed through send) at the top of the function for back-compat, but the queue is always empty in production. The wire-drive path is what makes notifications visible to the LLM: no fake user message, no meta prefix. **AED state transitions** (within `_run_loop`): transient provider failures first get up to three local retry attempts with 1/2/4s backoff, pending-tool-call healing, and `system.stuck_revive` retry messages, but without incrementing `aed_attempts` or rebuilding the session (`base_agent/turn.py:225-253`). Counted AED attempts then close pending tool calls with synthesized results (`base_agent/turn.py:255-265`), set STUCK (`base_agent/turn.py:267`) before rebuilding the session (`base_agent/turn.py:293-295`), and set ACTIVE again right before re-entering `_handle_message` with the recovery message (`base_agent/turn.py:298-299`). Without that ACTIVE flip, a recovered turn that does long work (file writes, mail, bash) leaves `.agent.json:state` reading "stuck" until the post-turn sweep at `base_agent/turn.py:300-322` finally writes `sleep_state` (IDLE) — visible to the TUI/portal as a stale "滞" badge while the agent is in fact busy. `_get_guard_limits` (`base_agent/turn.py:684`), `_process_response` (`base_agent/turn.py:774`; empty response raises at `base_agent/turn.py:809`).
- `base_agent/tools.py` — Tool surface: schemas, dispatch, registry (169 lines). `_dispatch_tool` (`base_agent/tools.py:14-33`), `_refresh_tool_inventory_section` (`base_agent/tools.py:36-50`), `_build_tool_schemas` (`base_agent/tools.py:53-107`) injects both `reasoning` and the reserved `secondary` schema property into eligible non-communication primary tools, `_add_tool` (`base_agent/tools.py:110-138`), `_remove_tool` (`base_agent/tools.py:141-149`), `_override_intrinsic` (`base_agent/tools.py:152-164`), `_has_capability` (`base_agent/tools.py:167-169`).
- `base_agent/prompt.py` — System prompt building, flushing, updating (55 lines). `_build_system_prompt` (`base_agent/prompt.py:9-16`), `_build_system_prompt_batches` (`base_agent/prompt.py:18-32`), `_flush_system_prompt` (`base_agent/prompt.py:34-42`), `_update_system_prompt` (`base_agent/prompt.py:44-55`).
- `base_agent/messaging.py` — Mail arrival, notification producers, outbound messaging (225 lines after notification migration and lock-safe system event merging). `_on_mail_received` (`base_agent/messaging.py:11-18`), `_on_normal_mail` (`base_agent/messaging.py:21-49`) — calls `_wake_nap` for sub-second sync latency and republishes the unread digest; **does NOT post `MSG_TC_WAKE` itself** — wake transitions are owned by the kernel notification sync mechanism. **`_rerender_unread_digest`** (`base_agent/messaging.py:52-106`) — email producer; calls `system.publish_notification(workdir, "email", header=…, icon="📧", data={count, newest_received_at, digest})` when count > 0, else `clear_notification(workdir, "email")`. **`_enqueue_system_notification`** (`base_agent/messaging.py:109-194`) — system events producer; merges `(event_id, source, ref_id, body, at)` into the events list inside `.notification/system.json`, capped at 20 newest. **Lazy `threading.Lock`** (`agent._system_notification_lock`) guards the read-modify-write merge so concurrent bursts (e.g. mail bounces) don't lose events. Only `system.json` needs the lock; `email.json` and `soul.json` recompute full state on every publish (no merge). Returns an `event_id` for back-compat with callers that expected a notif_id; per-id lifecycle is gone under the new model. `_notify` (`base_agent/messaging.py:197-204`), `_mail` (`base_agent/messaging.py:207-213`), `_send` (`base_agent/messaging.py:213-223`).

## Connections

- All submodules are leaves — they import nothing from `base_agent/`. Cross-module communication is mediated through the `agent` parameter (the BaseAgent instance).
- `__init__.py` imports from all submodules lazily (inside method bodies) to avoid circular imports at load time. Soul-flow pass-throughs now import from `intrinsics/soul/flow.py` and `intrinsics/soul/inquiry.py`.
- `__init__.py` → `notifications.py` (via `_sync_notifications` consuming `notification_fingerprint`/`collect_notifications`).
- `__init__.py` constructs `SessionManager` without notification-specific send hooks; notification delivery is owned by `_sync_notifications` plus the ACTIVE canonical-payload attach step in `turn.py`.
- `lifecycle.py` → `intrinsics/soul/flow.py` (via `agent._start_soul_timer()`, `agent._cancel_soul_timer()`), `intrinsics/soul/inquiry.py` (via `agent._run_inquiry()`), and `agent._sync_notifications()` on every heartbeat tick.
- `turn.py` → `intrinsics/soul/flow.py` (via `agent._cancel_soul_timer()`), `intrinsics/soul/inquiry.py` (via `agent._run_inquiry()`). The `tc_inbox` drain calls in `_handle_request`, the pre-request hook, and `_handle_tc_wake` are still wired but always find an empty queue under the `.notification/` redesign.
- `messaging.py` → `intrinsics/system` (via `publish_notification`/`clear_notification` for both `_rerender_unread_digest` and `_enqueue_system_notification`), and `notifications.collect_notifications` for the system-events RMW merge.
- `prompt.py` → `tools.py` (via `agent._refresh_tool_inventory_section()`).
- All submodules read agent attributes (`agent._config`, `agent._session`, `agent._chat`, `agent._notification_fp`, `agent._notification_live_holder`, etc.) — the agent object is the shared state.
- `messaging.py` imports `from ..message import _make_message, MSG_REQUEST, MSG_TC_WAKE` and `from ..i18n import t as _t`. (`MSG_TC_WAKE` is no longer posted from `messaging.py` itself; the kernel sync owns wake transitions.)
- `tools.py` imports `from ..intrinsics import ALL_INTRINSICS`, `from ..llm import FunctionSchema`, `from ..i18n import t as _t`.
- `turn.py` imports `from ..state import AgentState`, `from ..loop_guard import LoopGuard`, `from ..tool_executor import ToolExecutor`.
- `lifecycle.py` imports `from ..state import AgentState`, `from ..token_ledger import sum_token_ledger`.

## State

- `identity.py` mutates `.agent.json` (manifest) and `system/system.md` (identity prompt section) via `_build_manifest` (`base_agent/identity.py:58-97`) and `_update_identity` (`base_agent/identity.py:30-47`). The manifest now includes a sanitized `llm` block (`provider`, `model`, `base_url`) read from the live `LLMService` via `_safe_llm_from_service` (`base_agent/identity.py:100-177`). The wrapper subclass extends with `preset` (active/default/allowed) read from `init.json` and re-applies the LLM safelist for defense-in-depth — see `src/lingtai/agent.py`. The identity prose renderer (`base_agent/__init__.py:139-181` inside `_build_identity_section`) folds both blocks into the cached system-prompt prefix only when present, so bare BaseAgent instances stay quiet.
- `lifecycle.py` writes `.agent.heartbeat` (`base_agent/lifecycle.py:207-211`; the file's lifetime equals the Python process's lifetime — see "Heartbeat-file lifetime contract" above), consumes signal files (`.interrupt`, `.refresh`, `.suspend`, `.sleep`, `.prompt`, `.clear`, `.inquiry`, `.rules`), and reads `.notification/*.json` via `_sync_notifications` on every tick.
- `turn.py` writes `history/chat_history.jsonl` via `_save_chat_history` (delegated to `__init__.py`), writes `.status.json` and `logs/token_ledger.jsonl`, and mutates the live notification holder on dict-shaped tool results by adding/removing the canonical `notifications` + `_notification_guidance` payload. It also constructs the per-turn `ToolExecutor` with `working_dir=agent._working_dir` (`base_agent/turn.py:690-704`, `774-785`) so the executor can spill oversized tool results into `<workdir>/tmp/tool-results/` — see `tool_result_artifacts.py:spill_oversized_result`. Notification-pair injection in `__init__.py` bypasses `ToolExecutor` and therefore the preventive cap, by design. **Retroactive compaction (issue #144)**: before either AED retry branch, `_compact_history_before_retry` walks `_session.chat.interface._entries` and rewrites already-committed `ToolResultBlock.content` larger than the 5K retroactive cap into the same `lingtai_tool_result_spill` manifest shape — entries / ids / pairing / `synthesized` flag are preserved. The helper logs a single bounded summary event `aed_history_compacted` (`scanned_blocks`, `compacted_blocks`, `original_chars_total`, `replacement_chars_total`, up to 16 `artifact_paths`) and, when at least one block was rewritten, calls `_save_chat_history(ledger_source="retroactive_compaction")` so the persisted history matches the wire before the rebuilt session replays it. **Over-window routing**: errors whose `str(exc)` matches `_OVER_WINDOW_MSG_FRAGMENTS` (context-window / prompt-too-long / token-count phrasing across providers) short-circuit the transient retry loop and take the deterministic branch with `source="aed_over_window"`, since retrying on the same wire would refire the same error. **WorkerStillRunningError is exempt** — that branch puts the agent ASLEEP without touching ChatInterface, so compaction never runs (it would race the still-alive worker future). The helper itself is exception-safe; any failure logs `tool_result_compaction_failed` and never re-raises. A hard pre-send gate (run compaction whenever the wire exceeds a threshold *before* attempting send) is left as a TODO in `_run_loop`.
- `messaging.py` writes `.notification/email.json` (via `_rerender_unread_digest`) and merges into `.notification/system.json` (via `_enqueue_system_notification`, RMW under a per-agent `threading.Lock`).
- `__init__.py` writes the notification `(call, result)` pair into `history/chat_history.jsonl` via `_save_chat_history(ledger_source="notification_sync")` after a successful `_inject_notification_pair`; later holder moves preserve the pair but mutate its result dict into a skeleton placeholder. The injection event in `events.jsonl` carries `call_id`, `sources`, `summary`, **and `meta`** (the same `build_meta(agent)` dict — `current_time`, `context.{system_tokens,history_tokens,usage}`, `stamina_left_seconds`, `injection_seq` — that the agent itself sees in the synthesized tool-result body). The TUI's `parseEvent` reads the `meta` block and renders it as a faint footer line under the notification banner; older events.jsonl rows that pre-date this addition simply omit `meta` and the renderer drops the footer. See `lingtai/tui/internal/fs/session.go::parseEvent` notification branch and issue #40.

## Notes

- **`__init__.py` is large.** This is intentional — the constructor, properties, state machine, hooks, cross-cutting infrastructure (`_save_chat_history`, `_log`, notification sync, legacy tc_inbox drain), and pass-through stubs all live here. The package is not "thin shell + 6 leaves"; the remaining code is genuinely cross-cutting or bound to the class definition.
- **Pass-through pattern.** Each extracted method becomes a 2-line stub in `__init__.py` (lazy import + call). This preserves the `BaseAgent` class interface while the implementation lives in submodules. The `self` → `agent` conversion is mechanical but `_heartbeat_loop` (183 lines, 15+ cross-cluster calls) deserves extra-careful review.
- **Subclass overrides stay on `__init__.py`.** `_activate_preset`, `_activate_default_preset`, `_build_launch_cmd`, `_pre_request`, `_post_request`, `_on_tool_result_hook`, `_cpr_agent` are all overridden by the `Agent` subclass in the wrapper package. They must remain as methods on `BaseAgent`.
