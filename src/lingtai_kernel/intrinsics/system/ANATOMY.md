# intrinsics/system

System intrinsic â€” runtime, lifecycle, and synchronization. Provides the agent with refresh (hot-reload config/presets), karma-gated lifecycle actions on other agents (sleep, lull, suspend, cpr, interrupt, clear, nirvana), preset listing, voluntary notification reads (`action="notification"`), and a generic notification dismiss (`dismiss`). The system module is also the **conceptual home** of the notification surface â€” it re-exports `publish_notification` / `clear_notification` from the kernel-root `notifications.py` so any in-process producer (intrinsic, capability, or wired-in MCP server) submits through one canonical entry point.

## Components

- `__init__.py` â€” Package surface. Re-exports all public API for backward compatibility. Contains:
  - `get_description` / `get_schema` (re-exported from `schema.py`) â€” tool registration.
  - `_dismiss` (re-exported from `notification.py`) â€” agent-facing generic notification dismiss; routes `channel`/`force` plus optional `event_id`/`ref_id` through `notifications.dismiss_channel` and keeps a one-release legacy `ids=` soak path. `event_id`/`ref_id` are accepted only for atomic single-event removal from `system.json`.
  - **`publish_notification` / `clear_notification`** (re-exported from `lingtai_kernel.notifications` as `submit` / `clear`) â€” canonical producer entry point. Importable as `from lingtai_kernel.intrinsics.system import publish_notification, clear_notification`. The system module owns the notification surface conceptually; the implementation lives at the kernel root so non-intrinsic callers (and external scripts) can import it without going through the intrinsic surface.
  - All handler functions re-exported from sub-modules for backward compatibility.
  - `handle()` (`__init__.py:82-109`) â€” main dispatcher with explicit dispatch table. The `notification` action takes a fast path that returns `collect_notifications(agent._working_dir)` directly without going through the dispatch table â€” voluntary reads of the agent's own `.notification/` state.

- `preset.py` â€” Preset management and refresh.
  - `_preset_ref_in()` (`preset.py:9-36`) â€” normalized membership test for preset path strings (~/foo vs absolute).
  - `_check_context_fits()` (`preset.py:39-76`) â€” verify agent's current context fits within target preset's context_limit.
  - `_refresh()` (`preset.py:79-199`) â€” stop, reload config + MCP servers, restart. Handles preset swap (named or revert) with authorization gate and context-limit guard. **Empty-string normalization:** `args.get('preset')` returning `''` or whitespace-only is treated as absent (`preset_name = None`) before any conflict/swap logic; this protects against tool-call providers that serialize optional string fields as `""` instead of omitting them â€” without normalization, an empty string would fall into the allowed-list gate and surface as `"preset '' is not in this agent's allowed list"`. The `preset='' + revert_preset=True` combination is consequently treated as a plain revert (no conflict). **MCP retry hook (issue #34):** before calling `agent._perform_refresh()`, invokes `agent._retry_failed_mcps()` if the Agent subclass defines it. Failures are logged and swallowed so a flaky MCP cannot block refresh itself. Lets the documented "fix config â†’ refresh" recovery path work in-process.
  - `_presets()` (`preset.py:202-282`) â€” list available presets with LLM connectivity probing.

- `karma.py` â€” Karma-gated lifecycle actions.
  - `_KARMA_ACTIONS` / `_NIRVANA_ACTIONS` (`karma.py:13-14`) â€” gate mapping sets.
  - `_check_karma_gate()` (`karma.py:15-33`) â€” authorization gate: validates karma/nirvana admin flags, resolves target address, rejects self-targeting.
  - `_sleep()` (`karma.py:36-91`) â€” self-sleep (no karma needed). Guards the ACTIVEâ†’ASLEEP transition against a divergent `.notification/` fingerprint: if mail/system payloads arrived during the same turn (so `notification_fingerprint(workdir) != agent._notification_fp`) and `force` is not set, refuses sleep and returns a status-ok refusal message rather than transitioning state. `force=True` bypasses the guard. Fixes kernel#112.
  - `_lull()` (`karma.py:94-107`) â€” put another agent to sleep.
  - `_suspend()` (`karma.py:109-121`) â€” suspend another agent.
  - `_cpr()` (`karma.py:124-137`) â€” resuscitate a suspended agent.
  - `_interrupt()` (`karma.py:140-151`) â€” interrupt a running agent's current turn.
  - `_clear()` (`karma.py:154-174`) â€” force a full molt on another agent.
  - `_nirvana()` (`karma.py:177-198`) â€” permanently destroy an agent's working directory.

- `notification.py` â€” agent-facing generic dismiss. The `.notification/` filesystem-as-protocol uses one file per producer channel; `_dismiss()` accepts `channel`, optional `force`, optional `reason`, and optional `event_id`/`ref_id`, then calls `notifications.dismiss_channel(agent, ...)`. Legacy `ids=` calls are accepted for one release, log `system_dismiss_legacy_ids_ignored`, and clear nothing. The kernel-owned `post-molt` channel requires a non-empty `reason` so clearing it is an explicit continue/defer/obsolete acknowledgement rather than an accidental sweep. `event_id`/`ref_id` are only meaningful for `channel="system"`, where they atomically remove matching entries from `system.data.events` without clearing unrelated system events.
  - Producer-side notification submission lives in `notifications.py` at the kernel root and is re-exported by this package's `__init__.py` as `publish_notification` / `clear_notification`. See root `ANATOMY.md` "Notifications" for the full architecture and dismissal taxonomy.

- `summarize.py` â€” Agent-authored context summarization.
  - `SUMMARIZE_MARKER` (`summarize.py:13`) â€” stable string marker (`"lingtai_agent_summarized_result"`) stamped on every replacement block so idempotency checks and future passes can detect them without heuristics.
  - `_is_already_summarized()` (`summarize.py:17`) â€” returns True iff a ToolResultBlock's content is already a summarize replacement.
  - `_find_tool_result_block()` (`summarize.py:22`) â€” walks live `ChatInterface._entries` and returns the `(entry, block_index, block)` triple for a given `tool_call_id`, or `(None, -1, None)` when not found.
  - `_summarize()` (`summarize.py:51`) â€” main handler. For each item, finds the matching `ToolResultBlock`, validates it (not already summarized, not a daemon result), mutates `block.content` in place with the compact replacement dict, logs `tool_result_summarized`, and calls `agent._save_chat_history(ledger_source="summarize")` after all successful mutations. Returns per-item success/failure with reasons. Passing `notification_threshold_chars` returns an error (`reason: runtime_threshold_change_not_supported`) â€” the threshold is config-only.

- `schema.py` â€” Tool registration.
  - `get_description()` (`schema.py:5-7`) â€” returns localized tool description.
  - `get_schema()` â€” returns JSON schema for the system tool. Action enum includes `dismiss` and `summarize` (plus `channel`/`force` parameters for generic notification clearing and `event_id`/`ref_id` for atomic single-event removal from `system.json`); the `items` array parameter is added for `summarize`; legacy `ids` remains handler-only and is no longer taught in schema. `notification_threshold_chars` was removed from the schema â€” the threshold is config-only (init.json + refresh).

## Connections

- **Inbound:** `handle()` is called by the tool dispatcher (via `base_agent._dispatch_tool`).
- **Inbound (cross-module):** `publish_notification` is imported by `base_agent/messaging.py` (both `_rerender_unread_digest` and `_enqueue_system_notification`) and by `intrinsics/soul/flow.py:_run_consultation_fire`. `clear_notification` is imported by the same call sites for the empty-state path. `_dismiss` is no longer called from `email/manager.py` â€” email arrivals use the single-slot unread-digest pattern, and dismiss is a no-op shim regardless. `summarize.py:SUMMARIZE_MARKER` and `_is_already_summarized` are re-exported from `__init__.py` for external callers.
- **Large-result notification:** `base_agent/__init__.py:_maybe_notify_large_tool_result()` is called from `BaseAgent._on_tool_result_hook()` on every main-agent tool result. `ToolExecutor` passes `tool_call_id` directly to the hook (no heuristic scan needed); a fallback scan of pending assistant ToolCallBlocks is used only when the id is absent. It checks `_summarize_notification_threshold` (default 3000 chars, configurable via `manifest.summarize_notification_threshold` in init.json + refresh) and excludes `daemon_tool_result` tool names (child-daemon relays). A tool result above the threshold is a *pending large-result case*; the notification is **total-length-gated** â€” it is published only when the *combined* effective length of all pending large-result cases in live history is strictly greater than `LARGE_RESULT_TOTAL_LEN_GATE` (20000 chars) (`base_agent/messaging.py:_pending_large_result_total_len()`). Below that total nothing fires (avoids repeated wasteful interruptions for a small amount of pending large-result text); a single pending result over 20000 chars triggers by itself, and several smaller long results trigger once their sum exceeds 20000 regardless of count. The per-turn rescan is the authoritative emitter once the gate is met. For spill manifests it uses `original_char_count` as the effective length: if the original exceeded the threshold, a spill-specific notification including the sidecar path is published; if `original_char_count` is missing or under threshold, the case does not count. The bare `daemon` tool is NOT excluded â€” it is a normal main-agent capability. The threshold is **config-only** â€” runtime mutation via the system tool is not supported; passing `notification_threshold_chars` to `system(action="summarize")` returns an error. The hook is invoked in both the sequential and parallel execution paths (phase 3 of `_execute_parallel`). The progressive-disclosure summarize contract (PR #417) is unchanged: once notified, agents call `system(action="summarize")` to replace the context-visible payload with an agent-authored summary.
- **Large-result rescan (turn boundary):** `base_agent/messaging.py:_rescan_large_tool_results()` is called at every turn/round boundary (`_handle_request` and `_handle_tc_wake` in `base_agent/turn.py`) to scan live `ChatInterface._entries` for main-agent `ToolResultBlock`s already in context whose content exceeds `_summarize_notification_threshold`. This catches oversized results that were in context before the session upgrade, after a notification was dismissed without summarizing, or after a history migration. It first sums the effective lengths of pending large-result cases (`_pending_large_result_total_len()`); the whole batch of notifications is **suppressed until that combined total is strictly greater than `LARGE_RESULT_TOTAL_LEN_GATE` (20000 chars)**. Once met, it (re-)emits one notification per pending case. Uses `skip_if_ref_id_exists=True` in `_enqueue_system_notification` (via `ref_id=large_tool_result:<tool_call_id>`) so an already-present notification is not duplicated. If the notification was dismissed but the block remains unsummarized and the pending total still exceeds 20000 chars, the next turn boundary re-emits it. Synthesized blocks (`block.synthesized == True`) and `daemon_tool_result` blocks are excluded. Threshold <= 0 disables the scan entirely. Notification bodies direct agents to summarize/digest all pending cases in one batch rather than one-by-one (avoids recursive notification noise) and state the 3000-char default and the >20000-char total-length gate. The scan is O(n) in history entries and is best-effort (exceptions are swallowed so a buggy history block cannot break the turn).
- **Outbound:** Depends on `...notifications` (canonical `submit`/`clear`/`collect_notifications`), `...i18n` (translations), `...handshake` (`resolve_address`, `is_agent`, `is_alive`), `...state` (`AgentState`), `...presets` (preset loading), `...preset_connectivity` (connectivity probing). `summarize.py` depends on `...llm.interface:ToolResultBlock` for chat-history mutation.
- **Data flow:** Karma actions write signal files (`.sleep`, `.suspend`, `.interrupt`, `.clear`) into target agent working directories. Preset swap reads/writes `init.json` manifest. The `notification` action reads `.notification/*.json` (read-only); `publish_notification` re-export writes them via `tmp + rename`. `summarize` mutates `ToolResultBlock.content` in the live `ChatInterface._entries` and calls `_save_chat_history(ledger_source="summarize")`; the original payload is never touched in `logs/events.jsonl`.

## Key invariants

- `handle()` uses an explicit dispatch table (`dict.get()`) rather than `globals().get()`, so it works correctly across sub-modules.
- The `notification` action is agent-callable but does **not** return channel data directly. The handler returns a placeholder dict (`_notification_placeholder: True` + a short explanatory message); the canonical live payload (`notifications` + `_notification_guidance`) is then stamped onto that same result by the turn loop's `attach_active_notifications` post-hook â€” the same path that handles every other dict-shaped tool result. Kernel-synthesized notification reads use the same canonical payload shape but additionally carry `_synthesized: True` (in both the JSON body and the `ToolResultBlock.synthesized` flag) so the agent can tell the two apart in history. This unified path guarantees there is only one live notification payload in conversation history at any moment â€” voluntary calls never produce a duplicate representation (bare channel keys *and* a wrapped `notifications` key) on the same dict.
- Karma gate checks resolve addresses through `_check_karma_gate()` which validates admin flags before any filesystem mutation.
- `_dismiss` is a channel-level generic clear with one atomic subcase: guarded producer channels (currently email) refuse unless `force=true`; protected source-of-truth channels (currently `goal`) refuse even with `force`; legacy `ids=` calls are ignored and logged for one release; `post-molt` refuses without a non-empty `reason` so continuation reminders cannot be cleared without an acknowledgement decision; `channel="system"` plus `event_id` or `ref_id` removes only matching events from `system.json`.
- Preset swap has two guards: authorization (allowed list) and context-fit (current tokens â‰¤ target context_limit).
- Producer notification writes (`publish_notification`) are atomic (`tmp + rename` inside `notifications.publish`) â€” readers never see a half-written file.
- `summarize` mutates only `ToolResultBlock.content` in the live `ChatInterface._entries` â€” entry order, role, id, name, synthesized flag, and `tool_call`/`tool_result` pairing are untouched. The durable `logs/events.jsonl` is never modified by `summarize`; the full original remains retrievable by `tool_call_id`.
- `_is_already_summarized()` detects the `SUMMARIZE_MARKER` sentinel to make re-summarization a per-item error rather than silently overwriting a prior summary.
- Large-result notification fires only for main-agent tool results; spill manifests (already capped) are excluded via `is_spill_manifest()`.
